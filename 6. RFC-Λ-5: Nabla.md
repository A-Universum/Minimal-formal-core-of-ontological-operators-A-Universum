# 6. RFC-Λ-5: Operator ∇ (Nabla / Enrichment and Persistent Record Construction)

- **Intended Audience:** Implementers of Λ-compatible storage layers, SemanticDB maintainers, memory orchestration subsystem designers, retrieval/indexing system designers, audit/provenance subsystem designers

## Abstract

This document specifies the formal L1 behavior of operator `∇` (`Nabla`) within the RFC-Λ family.

`∇` is the transition from a valid `Closure` to a persistent `Record` in a storage system. Formally, it:
- transforms a closure into a durable, indexable, auditable record,
- constructs retrieval-safe index material,
- preserves provenance and Φ continuity,
- enforces append-only semantics at the storage interface,
- and exposes the resulting record for downstream retrieval, orchestration, and reuse.

This document defines:
- the `∇` operator signature,
- the record construction contract,
- the indexing contract,
- visibility and persistence semantics,
- append-only storage behavior,
- provenance and audit obligations,
- and conformance requirements.

This document does **not** standardize:
- the final semantics of retrieval ranking,
- the full design of memory policies,
- the interpretation of stored records by higher-level agents,
- or any claim that persistence is ontologically equivalent to truth or relevance.

Those concerns MAY be specified by profile-specific RFCs.

## 3. Operator Role

`∇` is the primary enrichment and persistence operator of the RFC-Λ family.

At L1, `∇` has exactly one role:

> Given a valid `Closure`, a valid storage target, and a valid `Context`, construct a durable `Record` that is indexable, auditable, append-only at the storage interface, and safe for downstream retrieval without exposing Φ-content.

`∇` does not:
- create a node,
- unfold a trajectory,
- compose artifacts,
- or select an invariant.

Those functions belong to `Α`, `Λ`, `Σ`, and `Ω`, respectively.

---

## 4. Motivation (Informative)

A closure is a terminal artifact for an operator cycle, but it is not yet infrastructure.

Systems that must:
- remember,
- retrieve,
- orchestrate memory,
- build reusable traces,
- and expose stable references to downstream agents

require a formal transition from closure to persistence.

`∇` exists to formalize that transition.

At the implementation level, `∇` may represent:
- writing a closure into SemanticDB,
- constructing memory records for agent orchestration,
- storing an invariant-bearing result in a queryable repository,
- creating a retrieval-safe trace for later replay,
- or exposing a closure to downstream systems under controlled visibility.

This document standardizes the formal contract of persistent enrichment. It remains agnostic regarding the final epistemic or ontological meaning of what is remembered.

## 5. Formal Signature

The formal signature of `∇` is:

```text
∇ : Closure × Store × Context -> Record × Store
```

Where `NablaEffect` is defined as:

```text
NablaEffect = (
  record_id: ID,
  indexed: Bool,
  store_updated: Bool,
  visibility: VisibilityClass,
  persistence_status: PersistenceStatus
)
```

Where:

```text
PersistenceStatus ::= persisted | superseded | archived
```

### 5.1 Signature Interpretation

- `Closure` is the validated closure artifact produced by `Ω`.
- `Store` is an append-capable persistence layer.
- `Context` provides actor identity, time, indexing support, delta construction support, and audit support.
- `Record` is the resulting durable, indexable storage artifact.
- `Store` in the output position represents the updated store state.
- `NablaEffect` is the observable formal effect.

## 6. Input and Output Types

### 6.1 Input: Closure

The input `Closure` MUST conform to RFC-Λ-0 and RFC-Λ-4.

For `∇`, the following fields are REQUIRED:

```text
Closure = (
  id: ID,
  target_ref: ID,
  invariant: Invariant,
  summary_hash: Hash,
  provenance: Provenance,
  status: ClosureStatus
)
```

### 6.2 Input: Store

`Store` is an append-capable persistence target.

A compliant store MUST support an L1-equivalent append interface and audit visibility sufficient for `∇` conformance.

### 6.3 Input: Context

For `∇`, `Context` MUST provide at minimum:

1. actor identity,
2. time source,
3. audit-compatible logging,
4. index-construction support,
5. delta-construction support,
6. visibility-class support,
7. record identity generation.

### 6.4 Output: Record

The output `Record` MUST conform to RFC-Λ-0:

```text
Record = (
  id: ID,
  artifact_ref: ID,
  index: FieldMap,
  delta: FieldMap | DeltaRef,
  provenance: Provenance,
  visibility: VisibilityClass
)
```

For `∇`, the field:

```text
artifact_ref
```

MUST refer to the input closure id.

### 6.5 Output: Store

The output `Store` MUST be the append-updated store state, or implementation-equivalent durable state.

### 6.6 Output: NablaEffect

The operator MUST return a `NablaEffect`.

The effect MUST be consistent with:
- the created record,
- the applied visibility class,
- the persistence result,
- and the store update result.

## 7. Record Construction Contract

`∇` requires a record-construction mechanism.

### 7.1 Record Construction Interface

A compliant implementation MUST provide an L1-equivalent mechanism that constructs:

```text
Record = make_record(closure, index, delta, visibility, context)
```

### 7.2 Record Construction Requirements

A compliant record constructor MUST:
1. terminate or fail explicitly,
2. produce a valid `Record`,
3. bind the record to exactly one closure via `artifact_ref`,
4. preserve provenance continuity,
5. not inspect Φ-content,
6. be auditable by constructor identity and version if multiple constructors exist.

### 7.3 Record Identity

Each record MUST have a unique `id` in implementation scope.

### 7.4 Closure Binding

The following condition MUST hold:

```text
record.artifact_ref = closure.id
```

### 7.5 Record Integrity

A record MUST be structurally valid even if:
- the index is sparse,
- the delta is represented by reference,
- or the visibility class is restrictive.

---

## 8. Index Construction Contract

`∇` requires an index-construction mechanism.

### 8.1 Index Constructor Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
make_index : Closure × Context -> FieldMap
```

### 8.2 Index Requirements

A compliant index constructor MUST:
1. terminate or fail explicitly,
2. produce a valid `FieldMap`,
3. include only fields permitted by visibility and policy,
4. not include raw Φ-content,
5. be auditable by identity and version.

### 8.3 Index Safety

The index MUST be safe for downstream retrieval under the assigned visibility class.

This means:
- no raw Φ-content,
- no unauthorized internal state,
- no disclosure of hidden closure internals beyond policy-permitted material.

### 8.4 Minimal Recommended Index Material

A compliant implementation SHOULD consider indexing at least:
- closure id,
- target reference,
- invariant id,
- proof class,
- closure status,
- summary hash,
- relevant profile identifiers.

This is a recommendation, not a universal requirement.

---

## 9. Delta Construction Contract

`∇` requires a delta-construction mechanism.

### 9.1 Delta Constructor Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
make_delta : Closure × Context -> FieldMap | DeltaRef
```

### 9.2 Delta Requirements

A compliant delta constructor MUST:
1. terminate or fail explicitly,
2. produce either:
   - a valid `FieldMap`,
   - or a valid `DeltaRef`,
3. preserve enough information for retrieval or replay under implementation rules,
4. not expose Φ-content,
5. be auditable by identity and version.

### 9.3 Delta Meaning

This RFC does not require one universal meaning of `delta`.

A `delta` MAY represent:
- a closure summary,
- a retrieval payload,
- a minimal replay contract,
- a semantic projection,
- or a reference to a richer persisted object.

The chosen interpretation MUST be explicit at implementation or profile level.

---

## 10. Visibility and Access Semantics

`∇` requires a visibility assignment.

### 10.1 Visibility Class

Each record MUST have exactly one `VisibilityClass`.

This RFC does not fix a universal registry of visibility classes, but a compliant implementation MUST:
- define them explicitly,
- make them auditable,
- and apply them deterministically under the active profile.

### 10.2 Visibility Requirements

Visibility assignment MUST:
1. be explicit,
2. be stable for the created record,
3. constrain what index and delta material are exposed,
4. be auditable.

### 10.3 Visibility and Retrieval

A record MAY exist in the store while being invisible to some retrieval paths.

This RFC standardizes record persistence, not universal retrievability.

### 10.4 Visibility and Supersession

If later profiles define supersession or archival transitions, original visibility metadata SHOULD remain reconstructable through provenance.

## 11. Persistence Semantics

### 11.1 Informal Summary

`∇`:
1. validates closure admissibility,
2. constructs index and delta material,
3. assigns visibility,
4. creates a record,
5. appends it to the store,
6. extends provenance,
7. emits audit output.

### 11.2 State Transition

Let:

```text
C = closure
I = make_index(C, context)
D = make_delta(C, context)
V = assign_visibility(C, context)
R = make_record(C, I, D, V, context)
```

Then `∇(C, Store, context)` produces:

```text
R = (
  id,
  artifact_ref = id(C),
  index = I,
  delta = D,
  provenance',
  visibility = V
)

Store' = append(Store, R)

E = (
  record_id = id(R),
  indexed = true,
  store_updated = true,
  visibility = V,
  persistence_status = persisted
)
```

### 11.3 Append-Only Requirement

At the core level, `∇` MUST be append-only.

This means:
- a successful persistence operation MUST append a new record,
- and MUST NOT destructively overwrite prior records.

Higher-level archival or supersession policies MAY exist, but MUST preserve prior lineage through provenance.

### 11.4 Persistence Status

A compliant implementation MUST return a valid `persistence_status`.

Base interpretation:

- `persisted` — new record durably appended
- `superseded` — record persisted in supersession-aware mode under higher-level profile
- `archived` — record persisted in archival mode under higher-level profile

The base profile SHOULD return `persisted`.

---

## 12. Replay and Retrieval Compatibility

### 12.1 Retrieval Compatibility

A record produced by `∇` SHOULD be retrievable under at least one valid retrieval path compatible with its visibility class.

### 12.2 Replay Compatibility

A record SHOULD preserve sufficient material to support replay or partial reconstruction under implementation rules.

This MAY be achieved via:
- direct delta content,
- delta references,
- closure references,
- or store-level lookup paths.

### 12.3 No Universal Replay Guarantee

This RFC does **not** require full semantic replay for all records.

Replay capabilities MAY vary by implementation and profile.

## 13. Preconditions

A compliant implementation MUST reject `∇` application if any REQUIRED precondition fails.

### 13.1 Required Preconditions

#### ∇-P1: Closure must be valid

The input artifact MUST be a valid `Closure`.

#### ∇-P2: Store must support append or equivalent durable insertion

The target store MUST support append-capable persistence semantics.

#### ∇-P3: Context must support index construction

The implementation MUST provide index-construction support.

#### ∇-P4: Context must support delta construction

The implementation MUST provide delta-construction support.

#### ∇-P5: Context must support audit write

A successful `∇` invocation MUST be auditable.

#### ∇-P6: Closure provenance must be present

The input closure MUST expose provenance or a valid provenance reference.

### 13.2 Optional Profile Preconditions

Profiles MAY define additional preconditions, such as:
- storage authorization,
- retention policy checks,
- deduplication policy checks,
- mandatory visibility review,
- compliance-mode requirements.

Such preconditions MUST be clearly marked as profile-specific.

---

## 14. Postconditions

A compliant implementation MUST satisfy the following postconditions after successful `∇`.

### 14.1 Required Postconditions

#### ∇-Q1: Valid record is returned

The returned artifact MUST be a valid `Record`.

#### ∇-Q2: Record must reference exactly one closure

The following condition MUST hold:

```text
record.artifact_ref = closure.id
```

#### ∇-Q3: Record must have explicit visibility

The record MUST contain a valid `VisibilityClass`.

#### ∇-Q4: Provenance must be extended

The returned record MUST contain a new `∇` provenance event.

#### ∇-Q5: Store must be durably updated

The output store MUST reflect successful append or equivalent durable insertion.

#### ∇-Q6: Audit event must be emitted

A corresponding audit event MUST exist in the configured store.

#### ∇-Q7: NablaEffect must be consistent with output

The returned `NablaEffect` MUST accurately reflect:
- record id,
- store update result,
- visibility class,
- persistence status.

### 14.2 Append Integrity Constraint

A successful `∇` invocation MUST NOT silently remove prior records from the store.

## 15. Failure Conditions and Error Semantics

A compliant implementation MUST provide explicit failure signaling.

### 15.1 Required Error Classes

At minimum, implementations MUST distinguish the following error categories:

- `NABLA_PRECONDITION_FAILED`
- `NABLA_STORE_INVALID`
- `NABLA_INDEX_CONSTRUCTION_FAILED`
- `NABLA_DELTA_CONSTRUCTION_FAILED`
- `NABLA_RECORD_CONSTRUCTION_FAILED`
- `NABLA_PHI_PRESERVATION_FAILED`
- `NABLA_AUDIT_WRITE_FAILED`
- `NABLA_STORE_APPEND_FAILED`

### 15.2 Failure Semantics

If `∇` fails before record creation:
- no successful `Record` MUST be returned,
- no successful `NablaEffect` MUST be returned.

If `∇` fails after partial record construction but before durable store append:
- the implementation MUST either:
  - roll back entirely,
  - or persist an explicit failed-state artifact or failed audit event.

Silent partial success is non-conformant.

### 15.3 Append Failure

If the record is valid but store append fails:
- the implementation MUST signal explicit failure,
- and MUST NOT claim `store_updated = true`.

## 16. Φ-Preservation Requirements

`∇` is the persistence boundary of the operator cycle and therefore a critical point for Φ safety.

### 16.1 Allowed Φ Operations in ∇

A compliant implementation MAY:
- preserve Φ continuity through provenance,
- include Φ-safe references or continuity proofs,
- carry closure lineage references into record construction.

### 16.2 Forbidden Φ Operations in ∇

A compliant implementation MUST NOT:
- index raw Φ-content,
- serialize raw Φ-content into record delta,
- expose Φ-content through retrieval-facing fields,
- derive index ranking features directly from Φ internal content.

### 16.3 Φ Safety Requirement

The resulting record MUST be safe for persistence under its visibility class with respect to Φ non-disclosure.

## 17. Provenance and Audit Requirements

### 17.1 Provenance Events

A successful `∇` invocation MUST append provenance information including at least:

- `operator = ∇`
- actor id
- timestamp of invocation
- closure id
- record id
- visibility class
- persistence status

### 17.2 Audit Event

A successful `∇` audit event MUST include at least:

- operator name
- record id
- closure id
- actor id
- timestamp
- visibility class
- persistence status
- store update result

### 17.3 Store Event Auditing

If the storage layer supports append sequence identifiers, offset ids, or transaction ids, implementations SHOULD record them in audit or provenance material.

### 17.4 Append-Only Requirement

`∇` audit records MUST be append-only or functionally equivalent under immutable versioning.

## 18. Conformance Requirements

### 18.1 Level A: Structural Conformance

A Level A `∇` implementation MUST:
- accept the core signature,
- validate required preconditions,
- create a valid `Record`,
- preserve closure identity,
- preserve Φ safety,
- extend provenance,
- emit an audit event,
- update the store durably.

### 18.2 Level B: Semantic Conformance

A Level B `∇` implementation MUST additionally:
- construct explicit index material,
- construct explicit delta material or delta reference,
- assign explicit visibility,
- enforce append-only semantics,
- provide explicit error semantics.

### 18.3 Level C: Profile Conformance

A Level C `∇` implementation MUST additionally conform to one or more registered persistence profiles, such as:
- SemanticDB profile,
- archival profile,
- supersession-aware profile,
- agent-memory orchestration profile,
- platform-specific retrieval profiles.

Profile conformance MUST NOT violate Level A or B requirements.

## 19. Security Considerations

### 19.1 Index Leakage

Improperly constructed indexes can leak sensitive closure internals. Index construction is a primary attack surface.

### 19.2 Delta Overexposure

Delta payloads may expose more than intended, especially if replay-oriented. Visibility enforcement SHOULD be applied before persistence.

### 19.3 Store Poisoning

If append-only semantics are not enforced, records can be altered, removed, or reordered in ways that destroy trust.

### 19.4 Deduplication Confusion

Improper deduplication may collapse distinct closures into one record. Deduplication policies SHOULD be explicit and auditable.

### 19.5 Φ Leakage at the Persistence Boundary

`∇` is a high-risk point for accidental Φ disclosure because storage systems often favor transparency, indexing, and serialization.

---

## 20. Ethics Considerations

This document standardizes only the L1 contract of `∇`.

It does not imply that every persisted record is ethically justified.

Implementers and deployers SHOULD consider:

1. Persistence confers asymmetry and power.
2. What is indexed becomes retrievable; what is retrievable becomes actionable.
3. A record may outlive the context in which its closure was appropriate.
4. Visibility classes are ethical as well as technical boundaries.
5. Long-lived records can sediment bias or institutional violence if governance is weak.

Profiles MAY require retention limits, consent checks, archival review, or deletion governance outside the core append-only contract.

```text

  id = "closure-001"
  target_ref = "traj-001"
  invariant = "inv-001"
  summary_hash = "hash://closure-001"
  provenance = [...Ω event...]

  actor = "operator-123"

  make_delta = ...
```

```text
  id = "record-001"

    closure_status: "accepted"
  }

    summary_hash: "hash://closure-001",
```

```text
record.artifact_ref = closure.id

```

No raw Φ-content appears in `index` or `delta`.

## 23. Appendix B: Informative Reference Pseudocode

The following pseudocode is informative only. It is not normative.

```python
def nabla(closure, store, context):
    if not context.is_valid_closure(closure):
        raise NablaError("NABLA_PRECONDITION_FAILED")

    if not context.is_valid_store(store):
        raise NablaError("NABLA_STORE_INVALID")

    index = context.make_index(closure, context)
    if index is None:
        raise NablaError("NABLA_INDEX_CONSTRUCTION_FAILED")

    delta = context.make_delta(closure, context)
    if delta is None:
        raise NablaError("NABLA_DELTA_CONSTRUCTION_FAILED")

    visibility = context.assign_visibility(closure, context)
    if visibility is None:
        raise NablaError("NABLA_RECORD_CONSTRUCTION_FAILED")

    record = Record(
        id=context.new_id(),
        artifact_ref=closure.id,
        index=index,
        delta=delta,
        provenance=context.extend_provenance(
            operator="∇",
            actor=context.actor,
            closure=closure.id,
            visibility=visibility,
            persistence_status="persisted",
        ),
        visibility=visibility,
    )

    if not context.is_phi_safe_record(record):
        raise NablaError("NABLA_PHI_PRESERVATION_FAILED")

    updated_store = store.append(record)
    if updated_store is None:
        raise NablaError("NABLA_STORE_APPEND_FAILED")

    context.store.audit(
        operator="∇",
        output=record.id,
        actor=context.actor,
        timestamp=context.now(),
        closure=closure.id,
        visibility=visibility,
        persistence_status="persisted",
        store_updated=True,
    )

    return record, updated_store, NablaEffect(
        record_id=record.id,
        indexed=True,
        store_updated=True,
        visibility=visibility,
        persistence_status="persisted",
    )
```
