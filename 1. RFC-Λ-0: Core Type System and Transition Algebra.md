## Abstract

This document defines the minimal formal kernel of the Λ operator system. It specifies the L1 formal layer required for interoperable implementations of the operators `Α`, `Λ`, `Σ`, `Ω`, and `∇`.

This document does **not** formalize the full ontological or phenomenological claims associated with the Λ-universe. Instead, it formalizes the executable interface: core data types, transition contracts, provenance requirements, Φ-preservation rules, and conformance criteria.

The purpose of this document is to provide a stable base for:
- reference implementations,
- platform-specific extensions,
- conformance testing,
- auditability,
- and versioned evolution of the RFC-Λ family.

## 1. Status of This Memo

This document is a **Draft** specification.

It may change without notice. Implementers SHOULD treat all interfaces defined here as provisional until the document reaches `Proposed` or `Accepted` status.

This document defines the **minimal** L1 formal kernel only. Higher-level operator-specific RFCs MAY extend this document but MUST NOT contradict its invariants.

---

## 2. Conventions and Normative Language

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, **MAY**, and **OPTIONAL** in this document are to be interpreted as described in RFC 2119 and RFC 8174.

Unless otherwise stated:

- `L0` refers to the explicitly non-formalizable layer.
- `L1` refers to the formalizable and testable layer.
- `L2` refers to implementation-specific code and optimizations.
- `Φ` refers to the opaque, non-interpretable residue preserved across operations.

Greek operator names are treated as protocol identifiers:
- `Α` (`Alpha`)
- `Λ` (`Lambda`)
- `Σ` (`Sigma`)
- `Ω` (`Omega`)
- `∇` (`Nabla`)
- `Φ` (`Phi`)

---

## 3. Scope

This document standardizes:

1. The core L1 type system.
2. The minimal transition algebra for Λ-compatible operators.
3. The required invariants for provenance, auditability, and Φ-preservation.
4. The minimal interoperability surface between operators and storage systems.

This document does **not** standardize:

1. Full ontological interpretation.
2. Phenomenological validation.
3. Human review procedures beyond formal gate presence.
4. Specific scoring functions for coherence, generativity, or emergence.
5. Platform-specific code generation or optimization details.
6. Final semantics for operator profiles such as `rho`, `sigma`, and `tau`.

Those concerns MAY be specified in subsequent RFCs.

---

## 4. Layer Model

The RFC-Λ family adopts a three-layer model.

```text
L0: Φ-Layer (non-formalizable)
- ontological depth
- phenomenological residue
- ethical surplus
- explicit incompleteness

L1: Λ-Layer (formalizable)
- syntax
- semantics
- state transitions
- conformance contracts
- persistence and audit requirements

L2: Implementation Layer
- platform-specific code
- optimization details
- vendor extensions
- deployment constraints
```

### 4.1 Layer Separation Requirements

An implementation MUST distinguish L1 obligations from L0 commentary.

An implementation MUST NOT claim L1 conformance based solely on L0 alignment.

An implementation MAY carry L0 metadata, but such metadata MUST NOT be treated as executable semantics unless separately specified in L1.

## 5. Design Principles

### 5.1 Minimal Formality

The core kernel MUST formalize only what is necessary for interoperability.

### 5.2 Explicit Incompleteness

Non-formalizable elements MUST be marked explicitly. They MUST NOT be smuggled into L1 as hidden assumptions.

### 5.3 Provenance Preservation

Every operator application MUST preserve and extend provenance.

### 5.4 Φ Preservation

Φ MUST be preserved as an opaque object or opaque reference. It MUST NOT be interpreted in L1.

### 5.5 Append-Only Auditability

Successful operator applications MUST be auditable through append-only records or equivalent immutable logs.

### 5.6 Non-Destructive Composition

Core operators MUST NOT silently destroy source artifacts.

## 6. Core Formal Domains

The following abstract domains are assumed.

### 6.1 Primitive Domains

- `Name` — field identifiers
- `Value` — field values
- `ID` — unique identifiers
- `Time` — timestamps
- `Hash` — cryptographic or integrity-preserving digest values
- `Bool` — boolean values

### 6.2 Derived Domains

#### 6.2.1 FieldMap

A `FieldMap` is a partial mapping:

```text
FieldMap = Name ⇀ Value
```

A `FieldMap` MAY be sparse. Absence of a field MUST NOT be conflated with nullity unless implementation semantics define such mapping explicitly.

#### 6.2.2 ΦToken

`ΦToken` is an opaque object.

Allowed L1 operations on `ΦToken` are restricted to:
- `hash(ΦToken) -> Hash`
- `preserve(ΦToken) -> ΦToken`
- `link(ΦToken) -> ΦRef`

No other L1 operation is permitted.

#### 6.2.3 Boundary

A `Boundary` is a total predicate over `FieldMap`:

```text
Boundary : FieldMap -> Bool
```

A `Boundary` determines whether a configuration is admissible within a formally defined region.

#### 6.2.4 Potential

A `Potential` is a candidate partial configuration:

```text
Potential = (id: ID, delta: FieldMap)
```

#### 6.2.5 Relation

A `Relation` is a typed link:

```text
Relation = (source: ID, label: Name, target: ID, weight: Value?)
```

#### 6.2.6 ConnectorType

A `ConnectorType` is a registered transition mode. The core registry initially defines:

- `↔` — bidirectional relation
- `→` — directed relation
- `⇢` — weak or defeasible relation

Implementations MAY register custom connector types.

## 7. Core Data Types

### 7.1 Vacuum

A `Vacuum` is the pre-marked domain for `Α`.

```text
Vacuum = (
  potentials: Set<Potential> | Stream<Potential>,
  phi: ΦToken,
  unmarked: Bool
)
```

#### Requirements

- `Α` MUST require `vacuum.unmarked = true`.
- `Vacuum` MUST carry a valid `ΦToken`.
- `Vacuum` MAY contain zero or more potentials.

---

### 7.2 Context

A `Context` provides execution capabilities.

```text
Context = (
  caps: Set<Name>,
  policies: Map<Name, Policy>,
  store: StoreRef,
  actor: ID,
  clock: TimeSource,
  connectors: Map<Name, Connector>
)
```

#### Requirements

- `Context` MUST expose capabilities used in precondition checks.
- `Context` MUST provide access to a store compatible with audit requirements.
- `Context` MUST identify the active actor.

---

### 7.3 Node

A `Node` is the primary marked artifact produced by `Α`.

```text
Node = (
  id: ID,
  fields: FieldMap,
  boundary: Boundary,
  phi_ref: ΦRef,
  provenance: Provenance,
  status: NodeStatus
)
```

`NodeStatus` values:
- `open`
- `closed`

#### Requirements

- A `Node` MUST have a valid `id`.
- A `Node` MUST have a `boundary`.
- A `Node` MUST carry a `phi_ref`, not raw `ΦToken` content.
- A `Node` MUST carry provenance.

---

### 7.4 Step

A `Step` is an element of a trajectory.

```text
Step = (
  id: ID,
  fields: FieldMap,
  boundary: Boundary,
  predecessor: ID | null,
  connector: ConnectorType | null,
  phi_ref: ΦRef,
  timestamp: Time
)
```

#### Requirements

- Every `Step` MUST be timestamped.
- Every `Step` MUST preserve `phi_ref` continuity.
- Every `Step` in a trajectory MUST be boundary-compatible with its governing origin or envelope boundary, as defined by operator semantics.

---

### 7.5 Trajectory

A `Trajectory` is the primary artifact produced by `Λ`.

```text
Trajectory = (
  id: ID,
  origin: Node,
  steps: Sequence<Step>,
  policy: PolicyName,
  status: TrajectoryStatus,
  provenance: Provenance
)
```

`TrajectoryStatus` values:
- `running`
- `suspended`
- `completed`
- `dead_end`
- `limit`

#### Requirements

- A `Trajectory` MUST reference a valid origin `Node`.
- `steps` MUST be temporally ordered.
- A `Trajectory` MUST record the policy name used.
- A `Trajectory` MUST maintain provenance continuity.

---

### 7.6 Composite

A `Composite` is the primary artifact produced by `Σ`.

```text
Composite = (
  id: ID,
  members: Set<ID>,
  relations: Set<Relation>,
  envelope: Boundary,
  phi_refs: Set<ΦRef>,
  provenance: Provenance,
  status: CompositeStatus
)
```

`CompositeStatus` values:
- `open`
- `closed`

#### Requirements

- A `Composite` MUST reference two or more member artifacts.
- A `Composite` MUST define an envelope boundary.
- `Σ` MUST NOT erase source member identity.

---

### 7.7 Invariant

An `Invariant` is a selected or candidate stabilized pattern.

```text
Invariant = (
  id: ID,
  schema: FieldMap | PatternRef,
  support: Set<ID>,
  stability: Value,
  proof_class: ProofClass
)
```

`ProofClass` values:
- `heuristic`
- `reproducible`
- `strong`

#### Requirements

- An `Invariant` MUST declare its `proof_class`.
- `reproducible` and `strong` invariants SHOULD expose machine-checkable or externally auditable stability criteria.

---

### 7.8 Closure

A `Closure` is the primary artifact produced by `Ω`.

```text
Closure = (
  id: ID,
  target_ref: ID,
  invariant: Invariant,
  summary_hash: Hash,
  provenance: Provenance,
  status: ClosureStatus
)
```

`ClosureStatus` values:
- `accepted`
- `rejected`
- `provisional`

#### Requirements

- A `Closure` MUST refer to exactly one target artifact.
- A `Closure` MUST contain exactly one selected invariant.
- A `Closure` MUST be immutable after acceptance, except through versioned supersession.

---

### 7.9 Record

A `Record` is the primary storage artifact produced by `∇`.

```text
Record = (
  id: ID,
  artifact_ref: ID,
  index: FieldMap,
  delta: FieldMap | DeltaRef,
  provenance: Provenance,
  visibility: VisibilityClass
)
```

#### Requirements

- A `Record` MUST refer to a previously created artifact.
- `Record.index` MUST contain only indexable and policy-permitted fields.
- `Record` MUST NOT expose Φ-content.

---

### 7.10 Policy

A `Policy` is a named selection profile used by `Λ`.

```text
Policy = (
  name: Name,
  next: SelectionFunction,
  halt: HaltFunction
)
```

Where:

```text
SelectionFunction : Step × Set<FieldMap> × Context -> FieldMap | SUSPEND | HALT | NONE
HaltFunction      : Trajectory × Context -> Bool
```

#### Requirements

- A `Policy` MUST have a unique registered name.
- Policy functions MAY be heuristic.
- Heuristic nature MUST be declared.
- A policy MUST NOT violate core boundary and provenance invariants.

## 8. Global Invariants

The following invariants apply to all compliant implementations.

### 8.1 Φ-Preservation Invariant

Every operator in the RFC-Λ family:

- MUST preserve Φ,
- MUST NOT inspect Φ-content,
- MUST maintain a valid `ΦRef` or equivalent preservation proof across transitions.

### 8.2 Provenance Monotonicity

For every successful operator application:

```text
provenance_out ⊇ provenance_in
```

No successful operator application MAY reduce provenance.

### 8.3 Auditability

Every successful operator application MUST create an audit event in the configured store.

### 8.4 Non-Silent Mutation

If an operator changes artifact state materially, the change MUST be represented either as:
- a new artifact, or
- a versioned mutation with explicit audit record.

Silent destructive overwrite is non-conformant.

### 8.5 Boundary Integrity

Whenever a boundary-governed artifact is extended or transformed, the responsible operator MUST define whether:
- the original boundary is preserved,
- an envelope boundary is created,
- or the operation is invalid.

This choice MUST be explicit.

## 9. Transition Algebra

The RFC-Λ core defines a minimal typed transition algebra over artifacts.

### 9.1 Artifact Classes

```text
Artifact ::= Vacuum | Node | Trajectory | Composite | Closure | Record
```

### 9.2 Primary Transition Forms

The core operators define the following transition classes:

```text
Α : Vacuum      -> Node
Λ : Node        -> Trajectory
Σ : Artifacts   -> Composite
Ω : Target      -> Closure
∇ : Closure     -> Record
```

Where:
- `Artifacts` is a finite set of two or more compatible artifacts
- `Target` is `Trajectory | Composite`

### 9.3 Canonical Sequences

The minimal canonical transition sequence is:

```text
Α -> Λ -> Ω -> ∇
```

The extended canonical transition sequence is:

```text
Α -> Λ -> Σ -> Ω -> ∇
```

### 9.4 Algebraic Properties

#### 9.4.1 Α Non-Idempotence

Repeated application of `Α` to the same vacuum descriptor MAY produce weakly equivalent nodes, but MUST NOT be assumed identity-preserving.

#### 9.4.2 Λ Policy Dependence

`Λ` is policy-dependent. Different policies MAY generate different trajectories from the same node.

#### 9.4.3 Σ Non-Commutativity

`Σ` MUST NOT be assumed commutative.

#### 9.4.4 Ω Idempotence on Accepted Closure

If an accepted closure already exists for a target under the same selector semantics, `Ω` SHOULD return the existing closure or a reference-equivalent closure.

#### 9.4.5 ∇ Append-Only Semantics

`∇` MUST be append-only at the core level.

## 10. Operator Signatures

This document defines only the minimal signatures and obligations.

### 10.1 Α (Alpha)

```text
Α : Vacuum × FieldMap × Context -> Node × AlphaEffect
```

#### Required Semantics

`Α` MUST:
1. validate that the vacuum is unmarked,
2. create a new node,
3. construct a boundary,
4. preserve Φ by reference,
5. record collapsed or excluded potentials where computable,
6. append an audit event.

#### Minimal Effect Type

```text
AlphaEffect = (
  created: Bool,
  collapsed_potential_ids: Set<ID>,
  phi_hash: Hash
)
```

---

### 10.2 Λ (Lambda)

```text
Λ : Node × Policy × Context -> Trajectory × LambdaEffect
```

#### Required Semantics

`Λ` MUST:
1. initialize a trajectory from a valid node,
2. generate candidate configurations,
3. apply a declared policy,
4. preserve origin-boundary compatibility unless a higher RFC explicitly defines alternative semantics,
5. preserve Φ across all steps,
6. append an audit event.

#### Minimal Effect Type

```text
LambdaEffect = (
  status: TrajectoryStatus,
  steps_count: Integer,
  invariant_candidates: Set<ID> | Set<PatternRef>,
  omega_ready: Bool
)
```

---

### 10.3 Σ (Sigma)

```text
Σ : Set<Artifact> × Relator × Context -> Composite × SigmaEffect
```

#### Required Semantics

`Σ` MUST:
1. require at least two source artifacts,
2. preserve source identities,
3. create explicit relations,
4. create an envelope boundary,
5. preserve Φ references from members,
6. append an audit event.

#### Minimal Effect Type

```text
SigmaEffect = (
  relation_count: Integer,
  invariant_candidates: Set<ID> | Set<PatternRef>,
  composition_status: Name
)
```

---

### 10.4 Ω (Omega)

```text
Ω : (Trajectory | Composite) × InvariantSelector × Context -> Closure × OmegaEffect
```

#### Required Semantics

`Ω` MUST:
1. target exactly one artifact,
2. select or reject a candidate invariant,
3. create an immutable closure if accepted,
4. compute a summary hash,
5. append an audit event.

#### Minimal Effect Type

```text
OmegaEffect = (
  closed: Bool,
  invariant_id: ID,
  proof_class: ProofClass,
  summary_hash: Hash
)
```

---

### 10.5 ∇ (Nabla)

```text
∇ : Closure × Store × Context -> Record × Store'
```

#### Required Semantics

`∇` MUST:
1. create an indexable record from a closure,
2. preserve provenance,
3. exclude Φ-content,
4. append to store,
5. append an audit event.

#### Minimal Effect Type

```text
NablaEffect = (
  record_id: ID,
  indexed: Bool,
  store_updated: Bool
)
```

## 11. Persistence and Provenance Model

### 11.1 Provenance

`Provenance` is an append-only chain or equivalent immutable structure that records:
- operator identifier,
- input references,
- output references,
- actor identifier,
- timestamp,
- optional review or approval metadata.

A compliant implementation MUST preserve provenance across all successful transitions.

### 11.2 Audit Store

A compliant implementation MUST expose or integrate with an append-only audit mechanism.

At minimum, each audit event MUST record:
- operator name,
- output artifact identifier,
- input artifact identifiers or references,
- timestamp,
- actor,
- status.

### 11.3 SemanticDB Compatibility

An implementation MAY use any backend store, but SemanticDB compatibility requires:
- append-only record semantics,
- immutable or versioned artifact references,
- query support over `Record.index`,
- non-disclosure of Φ-content.

## 12. Φ-Preservation Contract

The Φ contract is mandatory for all compliant implementations.

### 12.1 Allowed Operations

L1-compliant systems MAY:
- hash Φ,
- copy Φ opaquely,
- carry Φ by reference,
- compare Φ hashes where policy permits,
- attach Φ references to provenance.

### 12.2 Forbidden Operations

L1-compliant systems MUST NOT:
- deserialize Φ-content into field values,
- branch control logic on Φ internal content,
- expose Φ-content through logs, indexes, or debugging output,
- infer semantic content from Φ by undocumented side channels.

### 12.3 Preservation Proof

Implementations SHOULD provide one or more of:
- Φ hash continuity,
- Φ reference continuity,
- tamper-evident link chain,
- hardware-backed opaque handling.

### 12.4 Side-Channel Caution

This document does not guarantee protection from all side-channel leakage. Implementations handling sensitive Φ material SHOULD define additional mitigations.

## 13. Conformance Model

Conformance is defined at three levels.

### 13.1 Level A: Structural Conformance

A Level A implementation MUST:
- implement all core types or compatible equivalents,
- expose all core operator signatures,
- preserve Φ contract,
- preserve provenance,
- provide audit events.

### 13.2 Level B: Semantic Conformance

A Level B implementation MUST additionally satisfy:
- `Α` produces boundary-bearing nodes,
- `Λ` preserves declared boundary semantics,
- `Σ` preserves member identity and explicit relations,
- `Ω` produces immutable closures,
- `∇` produces indexable records without Φ exposure.

### 13.3 Level C: Profile Conformance

A Level C implementation MUST additionally support one or more registered policy profiles, such as:
- `rho`
- `sigma`
- `tau`

The exact scoring semantics of profiles MAY vary unless constrained by a profile-specific RFC. However, profile behavior MUST remain within the declared behavior class and MUST NOT violate core invariants.

## 14. Security Considerations

### 14.1 Provenance Forgery

If provenance can be forged, core trust collapses. Implementations SHOULD use tamper-evident logging, signed events, or equivalent mechanisms.

### 14.2 Φ Leakage

Implementations MUST treat Φ-handling as a sensitive operation. Debugging, serialization, and indexing paths are common leakage vectors.

### 14.3 Silent Mutation

Implementations MUST NOT allow operator effects to mutate prior artifacts without explicit versioning or replacement trace.

### 14.4 Policy Abuse

Heuristic `Λ` policies MAY be abused to create biased trajectories. Implementations SHOULD support audit visibility into policy identity and version.

### 14.5 Registry Poisoning

Custom connectors, policies, and selector profiles SHOULD be signed, versioned, and reviewable.

---

## 15. Ethics Considerations

This document standardizes L1 mechanics only. It does not claim that ethical adequacy can be reduced to conformance.

However, compliant implementations SHOULD account for the following:

1. Formal validity does not imply ethical legitimacy.
2. Boundary construction is never ethically neutral.
3. Composition MAY erase or distort difference if misused.
4. Closure MAY falsely present provisional patterns as final.
5. Enrichment MAY create durable institutional memory and therefore durable asymmetries.

Ethics review processes MAY be specified in higher-layer RFCs.

## 19. Appendix B: Minimal Canonical Transition Sequence

### B.1 Minimal Sequence

```text
Vacuum
  --Α-->
Node
  --Λ-->
Trajectory
  --Ω-->
Closure
  --∇-->
Record
```

### B.2 Extended Sequence

```text
Vacuum
  --Α-->
Node
  --Λ-->
Trajectory
  --Σ--> Composite
  --Ω-->
Closure
  --∇-->
Record
```

### B.3 Interpretation Constraint

The sequences above specify formal transition admissibility only. They do not by themselves establish ontological, ethical, or phenomenological validity.
