## Abstract

This document specifies the formal L1 behavior of operator `Σ` (`Sigma`) within the RFC-Λ family.

`Σ` is the transition from a finite set of admissible artifacts to a `Composite`. Formally, it constructs a composition from two or more artifacts by:
- preserving member identity,
- establishing explicit relations among them,
- constructing an envelope boundary for the resulting whole,
- preserving Φ references from all members,
- and recording provenance and audit information.

This document defines:
- the `Σ` operator signature,
- admissible input artifact classes,
- the `Relator` contract,
- the envelope boundary construction contract,
- composition semantics,
- provenance and audit obligations,
- and conformance requirements.

This document does **not** standardize:
- the final ontological meaning of synthesis, unity, or emergence,
- the full semantics of all relation labels,
- the final scoring semantics of all composition profiles,
- or any claim that every valid composition is meaningful, ethical, or generative.

Those concerns MAY be specified by profile-specific RFCs.

This document inherits:

Unless otherwise specified:

- and `Α` and `Λ` output semantics are those defined in RFC-Λ-1 and RFC-Λ-2.

## 3. Operator Role

`Σ` is the primary composition operator of the RFC-Λ family.

At L1, `Σ` has exactly one role:

> Given two or more admissible source artifacts, a declared `Relator`, and a valid `Context`, construct a `Composite` that preserves member identity while introducing explicit relations and an envelope boundary for the resulting whole.

`Σ` does not:
- create a `Node` from `Vacuum`,
- unfold an origin artifact into a `Trajectory`,
- finalize an invariant into a `Closure`,
- or persist enriched records into storage.

Those functions belong to `Α`, `Λ`, `Ω`, and `∇`, respectively.

---

## 4. Motivation (Informative)

A node can be valid. A trajectory can be valid. Yet neither by itself is sufficient to represent structured wholes composed of multiple artifacts in explicit relation.

`Σ` exists to formalize:
- composition,
- assembly,
- relation-bearing wholes,
- non-destructive synthesis,
- recursive integration,
- and the construction of envelope-level admissibility constraints.

At the implementation level, `Σ` may represent:
- graph-level composition of nodes,
- fusion of multiple trajectories into a higher-order structure,
- assembly of modules into a semantic aggregate,
- relational grouping of dialogue paths,
- or any other structured composition compatible with the core contract.

This document standardizes the formal contract of composition. It remains agnostic regarding the full ontological meaning of unity, emergence, reconciliation, or synthesis.

---

## 5. Formal Signature

The formal signature of `Σ` is:

```text
Σ : Set<Artifact> × Relator × Context -> Composite × SigmaEffect
```

Where `SigmaEffect` is defined as:

```text
SigmaEffect = (
  relation_count: Integer,
  invariant_candidates: Set<ID> | Set<PatternRef> | InvariantSummary,
  composition_status: CompositionStatus,
  envelope_ready: Bool
)
```

And:

```text
CompositionStatus ::= composed | partial | conflict
```

### 5.1 Signature Interpretation

- `Set<Artifact>` is a finite set of two or more admissible source artifacts.
- `Relator` is the declared relational-construction component.
- `Context` provides actor identity, relation support, envelope construction support, time, and storage.
- `Composite` is the resulting relation-bearing whole.
- `SigmaEffect` is the observable formal effect.

## 6. Admissible Input Artifact Classes

This document clarifies the admissible base input classes for `Σ`.

### 6.1 Base Artifact Classes for Σ

For the purposes of this RFC, the base admissible input domain is:

```text
Artifact ::= Node | Trajectory | Composite
```

### 6.2 Closure Exclusion in Base Profile

`Closure` is **not** part of the base admissible input domain for `Σ` in this RFC.

A future profile-specific RFC MAY define:
- `Closure`-aware composition,
- or composition over closed artifacts.

Such extensions MUST be explicit and versioned.

### 6.3 Cardinality Requirement

A `Σ` invocation MUST receive at least two source artifacts.

Formally:

```text
|Artifacts| >= 2
```

---

## 7. Input and Output Types

### 7.1 Input: Artifact Set

Each source artifact MUST be valid in its own type class:
- `Node`
- `Trajectory`
- or `Composite`

Each source artifact MUST provide:
- a stable identifier,
- provenance,
- and, where applicable, Φ references or equivalent inherited Φ continuity.

### 7.2 Input: Relator

`Relator` is a required input to `Σ`.

A compliant `Relator` MUST expose an L1-equivalent interface:

```text
Relator : Set<Artifact> × Context -> Set<Relation>
```

`Relator` is specified in detail in [Section 8](#8-relator-contract).

### 7.3 Input: Context

For `Σ`, `Context` MUST provide at minimum:

1. actor identity,
2. time source,
3. audit-compatible store,
4. envelope construction support,
5. relation-label and connector compatibility support,
6. optional invariant-candidate detection support.

### 7.4 Output: Composite

The output `Composite` MUST conform to RFC-Λ-0:

```text
Composite = (
  id: ID,
  members: Set<ID>,
  relations: Set<Relation>,
  envelope: Boundary,
  phi_refs: Set<ΦRef>,
  provenance: Provenance,
  status: CompositeStatusRFC0
)
```

Where `CompositeStatusRFC0` remains:

- `open`
- `closed`

For `Σ`, the created composite status MUST be:

```text
open
```

unless a higher-level profile explicitly defines immediate closed-composite semantics.

### 7.5 Output: SigmaEffect

The operator MUST return a `SigmaEffect`.

If invariant candidates cannot be detected, the implementation MUST still return:
- a valid `relation_count`,
- a valid `composition_status`,
- a valid `envelope_ready` flag,
- and an empty or deferred invariant candidate representation.

## 8. Relator Contract

`Σ` requires a relational-construction component.

### 8.1 Relator Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
Relator : Set<Artifact> × Context -> Set<Relation>
```

### 8.2 Relator Requirements

A compliant `Relator` MUST:
1. terminate or fail explicitly,
2. return zero or more explicit `Relation` instances,
3. preserve source artifact identity,
4. not mutate source artifacts,
5. not inspect Φ-content,
6. be auditable by identity and version.

### 8.3 Relation Validity

A returned `Relation` is valid if:
1. `source` refers to an input artifact member or a valid sub-member under profile-defined semantics,
2. `target` refers to an input artifact member or a valid sub-member under profile-defined semantics,
3. `label` is syntactically valid,
4. optional weight fields, if present, satisfy implementation value constraints.

### 8.4 Empty Relation Set

A `Relator` MAY return an empty relation set.

If so:
- `Σ` MAY still produce a valid `Composite`,
- but `SigmaEffect.composition_status` MUST reflect this appropriately, typically as `partial`.

An implementation MUST NOT silently claim `composed` if no explicit relations exist unless a higher-level profile defines a relation-free composition mode.

### 8.5 Determinism

A `Relator` SHOULD be deterministic for the same input artifact set and profile version.

If nondeterministic, this MUST be declared in metadata and SHOULD be auditable.

---

## 9. Envelope Boundary Construction Contract

`Σ` requires an envelope boundary constructor.

### 9.1 Envelope Constructor Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
make_envelope_boundary : Set<Artifact> × Set<Relation> × Context -> Boundary
```

### 9.2 Envelope Constructor Requirements

The envelope constructor MUST:
1. terminate or fail explicitly,
2. return a total predicate over `FieldMap`,
3. be auditable by identity and version,
4. not inspect Φ-content.

### 9.3 Envelope Validity

A boundary is considered a valid envelope boundary if:
1. it is callable or equivalent as a predicate over `FieldMap`,
2. it can be serialized, hashed, or version-referenced for provenance purposes,
3. it does not violate implementation-defined safety constraints.

### 9.4 Envelope Semantics

This RFC does **not** require that the envelope boundary be:
- a conjunction of member boundaries,
- a union of member boundaries,
- or any other fixed algebraic form.

The exact envelope construction strategy is implementation- or profile-defined.

However, the chosen strategy MUST be:
- explicit,
- versioned,
- and auditable.

### 9.5 Envelope Readiness

A successful `Σ` invocation MUST return:

```text
envelope_ready = true
```

if and only if a valid envelope boundary exists for the created composite.

## 10. Composition Semantics

### 10.1 Informal Summary

`Σ`:
1. validates the admissibility of source artifacts,
2. computes explicit relations,
3. constructs an envelope boundary,
4. creates a new composite artifact,
5. preserves member identity and inherited provenance,
6. preserves Φ references from all members,
7. emits provenance and audit records.

### 10.2 State Transition

Let:

```text
A = {a1, a2, ..., an}, n >= 2
R = relator(A, context)
BΣ = make_envelope_boundary(A, R, context)
```

Then `Σ(A, relator, context)` produces:

```text
C = (
  id,
  members = {id(a1), id(a2), ..., id(an)},
  relations = R,
  envelope = BΣ,
  phi_refs = ΦRefs(A),
  provenance',
  open
)

E = (
  relation_count = |R|,
  invariant_candidates = IC,
  composition_status = S,
  envelope_ready = true
)
```

### 10.3 Member Identity Preservation

`Σ` MUST preserve source member identity.

This means:
- source artifacts MUST remain addressable by their original identifiers,
- source artifacts MUST NOT be silently rewritten as a side effect of composition,
- and the composite MUST reference them explicitly.

### 10.4 Non-Destructive Composition

`Σ` MUST be non-destructive at the core level.

It MUST NOT:
- overwrite source artifacts,
- merge source artifacts into an untraceable blob,
- or discard their provenance.

### 10.5 Composition Status

A compliant implementation MUST return a valid `composition_status`.

Recommended interpretation:

- `composed` — explicit relations exist and envelope construction succeeded.
- `partial` — envelope exists, but relation structure is sparse, incomplete, or relation-free.
- `conflict` — composition was structurally created, but relation or compatibility analysis detected unresolved contradiction under implementation rules.

A profile-specific RFC MAY strengthen these semantics.

### 10.6 Relation Visibility

The resulting `Composite.relations` MUST be explicit and auditable.

A `Composite` is not merely a bag of members; it MUST carry the relation set produced by the `Relator`, even if that set is empty.

## 11. Recursive Composition Semantics

### 11.1 Base Rule

Because `Composite` is an admissible input artifact class for `Σ`, recursive composition is permitted.

Formally:

```text
Composite ∈ Artifact
```

for purposes of `Σ` input.

### 11.2 Recursive Validity

A recursive composition is valid if:
1. each input composite is itself valid,
2. provenance is preserved,
3. member identity remains traceable,
4. the new envelope boundary is successfully constructed.

### 11.3 No Implicit Flattening

This RFC does **not** require that nested composites be flattened.

An implementation MAY:
- preserve hierarchical structure,
- flatten recursively,
- or expose both views.

However, the chosen strategy MUST be explicit and auditable.

### 11.4 Provenance in Recursive Composition

Recursive composition MUST preserve enough information to reconstruct:
- immediate members,
- and the lineage of prior compositions.

## 12. Invariant Candidate Handling

`Σ` MAY detect invariant candidates during composition.

### 12.1 Optionality

Invariant candidate detection is OPTIONAL at the core level.

A compliant implementation MAY return:
- exact candidate identifiers,
- pattern references,
- summaries,
- or an empty set.

### 12.2 Candidate Detector Interface

If implemented, invariant candidate detection SHOULD expose an auditable mechanism equivalent to:

```text
detect_invariant_candidates : Composite × Context -> Set<ID> | Set<PatternRef> | InvariantSummary
```

### 12.3 Relationship to Ω

`SigmaEffect.invariant_candidates` creates the bridge from composition to closure (`Ω`).

This RFC does not require that every composite be `Ω`-ready.

However, `envelope_ready = true` is a necessary but not sufficient condition for later closure in most profiles.

## 13. Preconditions

A compliant implementation MUST reject `Σ` application if any REQUIRED precondition fails.

### 13.1 Required Preconditions

#### Σ-P1: At least two artifacts must be provided

```text
|Artifacts| >= 2
```

MUST hold.

#### Σ-P2: All source artifacts must be valid

Each source artifact MUST be valid in its declared type class.

#### Σ-P3: Relator must be declared

A valid `Relator` MUST be supplied.

#### Σ-P4: Context must support envelope construction

The implementation MUST provide an envelope boundary construction mechanism.

#### Σ-P5: Context must support audit write

A successful `Σ` invocation MUST be auditable.

#### Σ-P6: Source provenance must be present

Each source artifact MUST expose provenance or a valid provenance reference.

### 13.2 Optional Profile Preconditions

Profiles MAY define additional preconditions, such as:
- relation-policy approval,
- domain-specific composition authorization,
- anti-cycle constraints,
- type-homogeneity rules.

Such preconditions MUST be clearly marked as profile-specific.

---

## 14. Postconditions

A compliant implementation MUST satisfy the following postconditions after successful `Σ`.

### 14.1 Required Postconditions

#### Σ-Q1: Valid composite is returned

The returned artifact MUST be a valid `Composite`.

#### Σ-Q2: Member identities must be preserved

All source artifacts MUST remain explicitly referenceable in the composite.

#### Σ-Q3: Relations must be explicit

The resulting composite MUST include an explicit relation set, even if empty.

#### Σ-Q4: Envelope boundary must exist

The resulting composite MUST contain a valid envelope boundary.

#### Σ-Q5: Φ references must be preserved

The composite MUST preserve Φ references from source members as a set or equivalent aggregate structure.

#### Σ-Q6: Provenance must be extended

The returned composite MUST contain a new `Σ` provenance event.

#### Σ-Q7: Audit event must be emitted

A corresponding audit event MUST exist in the configured store.

#### Σ-Q8: SigmaEffect must be consistent with output

The returned `SigmaEffect` MUST accurately reflect:
- relation count,
- composition status,
- and envelope readiness.

### 14.2 Envelope Existence Constraint

The returned composite MUST satisfy:

```text
composite.envelope != null
```

and `SigmaEffect.envelope_ready` MUST be consistent with that fact.

## 15. Failure Conditions and Error Semantics

A compliant implementation MUST provide explicit failure signaling.

### 15.1 Required Error Classes

At minimum, implementations MUST distinguish the following error categories:

- `SIGMA_PRECONDITION_FAILED`
- `SIGMA_RELATOR_INVALID`
- `SIGMA_RELATION_CONSTRUCTION_FAILED`
- `SIGMA_ENVELOPE_CONSTRUCTION_FAILED`
- `SIGMA_PHI_PRESERVATION_FAILED`
- `SIGMA_AUDIT_WRITE_FAILED`
- `SIGMA_MEMBER_INVALID`

### 15.2 Failure Semantics

If `Σ` fails before composite creation:
- no successful `Composite` MUST be returned,
- no successful `SigmaEffect` MUST be returned.

If `Σ` fails after partial composition but before durable commit:
- the implementation MUST either:
  - roll back entirely,
  - or persist an explicit failed-state artifact or failed audit event.

Silent partial success is non-conformant.

### 15.3 Conflict vs Failure

`composition_status = conflict` is **not** equivalent to operator failure.

A `conflict` result may still be a valid successful output if:
- a composite was created,
- envelope construction succeeded,
- provenance and audit obligations were met.

## 16. Φ-Preservation Requirements

`Σ` is the first operator in the RFC-Λ family that may aggregate multiple Φ lineages.

### 16.1 Allowed Φ Operations in Σ

A compliant implementation MAY:
- collect Φ references from members,
- preserve them as a set,
- verify continuity,
- log continuity proofs,
- hash or compare Φ references where policy permits.

### 16.2 Forbidden Φ Operations in Σ

A compliant implementation MUST NOT:
- inspect Φ-content from any member,
- derive relation logic from Φ internal content,
- serialize Φ-content into relations, provenance, or audit records.

### 16.3 Φ Aggregation Requirement

For each successful composite:

```text
composite.phi_refs = aggregate_phi_refs(source_artifacts)
```

or an implementation-equivalent continuity-preserving aggregate relation.

This aggregate MUST preserve plurality.  
It MUST NOT collapse multiple Φ references into a single interpreted semantic object at L1.

## 17. Provenance and Audit Requirements

### 17.1 Provenance Events

A successful `Σ` invocation MUST append provenance information including at least:

- `operator = Σ`
- actor id
- timestamp of invocation
- source artifact ids
- relator identity
- envelope constructor identity
- final composition status

### 17.2 Provenance Aggregation

The resulting composite MUST preserve enough provenance to reconstruct:
- direct source members,
- operator identity,
- relation-construction context,
- and envelope-construction context.

### 17.3 Audit Event

A successful `Σ` audit event MUST include at least:

- operator name
- composite id
- actor id
- timestamp
- member count
- relation count
- composition status
- envelope readiness

### 17.4 Append-Only Requirement

`Σ` audit records MUST be append-only or functionally equivalent under immutable versioning.

## 18. Conformance Requirements

### 18.1 Level A: Structural Conformance

A Level A `Σ` implementation MUST:
- accept the core signature,
- validate required preconditions,
- create a valid `Composite`,
- preserve source member identity,
- preserve Φ references,
- extend provenance,
- emit an audit event.

### 18.2 Level B: Semantic Conformance

A Level B `Σ` implementation MUST additionally:
- expose explicit relations,
- construct a valid envelope boundary,
- distinguish `composed`, `partial`, and `conflict`,
- provide explicit error semantics.

### 18.3 Level C: Profile Conformance

A Level C `Σ` implementation MUST additionally conform to one or more registered composition profiles, such as:
- relation-dense composition,
- conflict-explicit composition,
- recursive hierarchical composition,
- or domain-specific composite profiles.

Profile conformance MUST NOT violate Level A or B requirements.

## 19. Security Considerations

### 19.1 Relation Manipulation

A malicious or biased `Relator` can create asymmetry while appearing formally valid. Implementations SHOULD version and audit relators.

### 19.2 Envelope Manipulation

A compromised envelope constructor can create misleading admissibility regions. Envelope-construction identity and version SHOULD be auditable.

### 19.3 Hidden Flattening

Recursive compositions that silently flatten structure can erase traceability. Implementations SHOULD make flattening behavior explicit.

### 19.4 Φ Leakage

Aggregating multiple Φ references increases leakage risk. Implementations SHOULD isolate Φ aggregation from debugging and serialization paths.

### 19.5 Provenance Collapse

If member provenance is not preserved, composites become opaque and unverifiable. Provenance collapse is a major integrity risk.

---

## 20. Ethics Considerations

This document standardizes only the L1 contract of `Σ`.

It does not imply that every formally valid synthesis is ethically justified.

Implementers and deployers SHOULD consider:

1. Composition can conceal domination under the appearance of unity.
2. Relation labels are not ethically neutral.
3. Envelope construction can erase or subordinate local boundaries.
4. Recursive composition can produce hierarchy while appearing merely structural.
5. A formally valid whole may still be a coercive whole.

Profiles MAY require human review, consent checks, or asymmetry audits for sensitive domains.

## 22. Appendix A: Canonical Minimal Example

### A.1 Input

```text
Artifact Set:
  A1 = Node(id="node-001", fields={topic: "A"}, ...)
  A2 = Trajectory(id="traj-001", origin="node-010", ...)

Relator:
  relator(A1, A2, context) returns:
    {
      (source="node-001", label="→", target="traj-001")
    }

Envelope Constructor:
  make_envelope_boundary({A1, A2}, relations, context) returns BΣ
```

### A.2 Output

```text
Composite:
  id = "comp-001"
  members = {"node-001", "traj-001"}
  relations = {
    ("node-001", "→", "traj-001")
  }
  envelope = BΣ
  phi_refs = {"phi://ref-node-001", "phi://ref-traj-001"}
  provenance = [...Σ event...]
  status = open

SigmaEffect:
  relation_count = 1
  invariant_candidates = {}
  composition_status = composed
  envelope_ready = true
```

### A.3 Required Truth Conditions

```text
|members| = 2
relation_count = 1
envelope_ready = true
```

Source artifact identities remain unchanged and referenceable.

## 23. Appendix B: Informative Reference Pseudocode

The following pseudocode is informative only. It is not normative.

```python
def sigma(artifacts, relator, context):
    if artifacts is None or len(artifacts) < 2:
        raise SigmaError("SIGMA_PRECONDITION_FAILED")

    for artifact in artifacts:
        if not context.is_valid_artifact(artifact):
            raise SigmaError("SIGMA_MEMBER_INVALID")

    if relator is None:
        raise SigmaError("SIGMA_RELATOR_INVALID")

    relations = relator(artifacts, context)
    if relations is None:
        raise SigmaError("SIGMA_RELATION_CONSTRUCTION_FAILED")

    envelope = context.make_envelope_boundary(artifacts, relations, context)
    if envelope is None:
        raise SigmaError("SIGMA_ENVELOPE_CONSTRUCTION_FAILED")

    phi_refs = context.aggregate_phi_refs(artifacts)
    if phi_refs is None:
        raise SigmaError("SIGMA_PHI_PRESERVATION_FAILED")

    member_ids = {artifact.id for artifact in artifacts}

    composition_status = "composed"
    if len(relations) == 0:
        composition_status = "partial"

    if context.detect_conflict(artifacts, relations, envelope):
        composition_status = "conflict"

    composite = Composite(
        id=context.new_id(),
        members=member_ids,
        relations=relations,
        envelope=envelope,
        phi_refs=phi_refs,
        provenance=context.extend_provenance(
            operator="Σ",
            actor=context.actor,
            members=member_ids,
            relator=context.relator_id(relator),
            envelope_constructor=context.envelope_profile_id(),
            composition_status=composition_status,
        ),
        status="open",
    )

    invariant_candidates = (
        context.detect_invariant_candidates(composite)
        if context.supports_invariant_detection()
        else set()
    )

    context.store.audit(
        operator="Σ",
        output=composite.id,
        actor=context.actor,
        timestamp=context.now(),
        member_count=len(member_ids),
        relation_count=len(relations),
        composition_status=composition_status,
        envelope_ready=True,
    )

    return composite, SigmaEffect(
        relation_count=len(relations),
        invariant_candidates=invariant_candidates,
        composition_status=composition_status,
        envelope_ready=True,
    )
```
