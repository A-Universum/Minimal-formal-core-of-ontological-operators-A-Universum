## Abstract

This document defines the formal L1 contract for `Φ` (`Phi`) within the RFC-Λ family.

`Φ` is not a constructive operator. It is a **cross-cutting preservation contract** governing how opaque, non-interpretable residues are carried through all RFC-Λ transitions.

This document specifies:
- the `ΦToken`, `ΦRef`, and `ΦHash` abstractions,
- the allowed and forbidden L1 operations on Φ-bearing objects,
- continuity requirements across `Α`, `Λ`, `Σ`, `Ω`, and `∇`,
- provenance and audit obligations,
- leakage classes and minimum mitigation expectations,
- and conformance levels for Φ-safe implementations.

This document does **not** define:
- the internal structure or semantic content of Φ,
- any L0 interpretation of Φ,
- or any authorized mechanism for interpreting Φ at L1.

The purpose of this specification is to make non-inspection and opaque preservation explicit, testable, and enforceable at the formal layer.

## 3. Scope

This document standardizes the L1 handling of opaque preservation.

Specifically, this document standardizes:
1. the opaque status of `ΦToken`,
2. the permissible L1 operations on Φ-bearing structures,
3. the continuity obligations across the operator cycle,
4. provenance and audit obligations related to Φ,
5. minimum leakage detection and response requirements,
6. conformance levels for Φ-safe systems.

This document does **not** standardize:
1. the generation semantics of Φ at L0,
2. the semantic interpretation of Φ,
3. any “meaning extraction” interface for Φ,
4. any operator that transforms Φ-content into L1-visible content,
5. philosophical or theological claims about what Φ “is”.

---

## 4. Role of Φ in the RFC-Λ Family

`Φ` is the formal marker of non-inspectable continuity across the RFC-Λ family.

At L1, `Φ` serves four functions:

1. **Opacity Preservation**  
   It guarantees that certain content remains uninterpretable within the L1 contract.

2. **Continuity Binding**  
   It allows artifacts across the operator cycle to carry provable opaque lineage.

3. **Reduction Resistance**  
   It prevents L1-compliant implementations from collapsing all meaningful state into inspectable fields.

4. **Boundary Declaration**  
   It marks the limit of what RFC-Λ standardizes formally.

`Φ` is therefore not a semantic payload in L1. It is a preservation boundary.

## 5. Core Types

### 5.1 ΦToken

`ΦToken` is an opaque object.

```text
ΦToken ::= opaque
```

Its internal structure is intentionally unspecified at L1.

A compliant L1 implementation MUST treat `ΦToken` as non-interpretable.

---

### 5.2 ΦRef

`ΦRef` is a stable opaque reference to a `ΦToken`.

```text
ΦRef ::= ref(ΦToken)
```

A `ΦRef` MAY be:
- a pointer,
- a capability reference,
- a handle,
- a lineage token,
- or another implementation-defined opaque reference form.

Its concrete representation is not standardized by this RFC.

---

### 5.3 ΦHash

`ΦHash` is an integrity-preserving digest derived from a `ΦToken`.

```text
ΦHash ::= hash(ΦToken)
```

A `ΦHash` MAY be used for:
- continuity checks,
- audit correlation,
- tamper evidence,
- and lineage support.

A `ΦHash` MUST NOT be treated as a semantic surrogate for Φ-content.

---

### 5.4 ΦContinuityProof

A `ΦContinuityProof` is any implementation-defined auditable structure sufficient to support the claim that two artifacts belong to the same opaque lineage without exposing Φ-content.

This RFC does not mandate one universal representation.

A compliant implementation MAY realize continuity proof using:
- provenance chains,
- hash lineage,
- signed reference transitions,
- capability continuity,
- or profile-specific proof artifacts.

---

## 6. Allowed and Forbidden Operations

### 6.1 Allowed L1 Operations

A compliant L1 implementation MAY perform the following operations on Φ-bearing objects:

- `preserve(ΦToken) -> ΦToken`
- `link(ΦToken) -> ΦRef`
- `hash(ΦToken) -> ΦHash`
- `compare(ΦHash, ΦHash) -> Bool`
- `carry(ΦRef) -> ΦRef`
- `validate_continuity(...) -> Bool` using provenance- or proof-based mechanisms

### 6.2 Restricted Operations

The following operations are conditionally allowed only if they do **not** reveal Φ-content:

- storing `ΦRef` in provenance
- storing `ΦHash` in audit logs
- using `ΦHash` for integrity checks
- using `ΦRef` as a lineage handle in non-public metadata

### 6.3 Forbidden L1 Operations

A compliant L1 implementation MUST NOT:

1. inspect the internal content of `ΦToken`;
2. deserialize `ΦToken` into L1-visible fields;
3. branch control flow on Φ internal content;
4. derive policy, selector, ranking, or visibility decisions from Φ internal content;
5. log raw `ΦToken` content;
6. serialize raw `ΦToken` into storage, index, or audit fields;
7. expose raw `ΦToken` through debugging, telemetry, or query interfaces;
8. use `ΦHash` as if it were interpreted content;
9. infer semantic properties from implementation-side accidental leakage where that inference is intentionally encoded into L1 behavior.

### 6.4 Prohibition on Semantic Rebinding

A compliant implementation MUST NOT treat:
- `ΦRef`,
- `ΦHash`,
- or continuity metadata

as a substitute semantic representation of Φ.

They are identity/continuity artifacts only.

## 7. Continuity Model

### 7.1 Basic Principle

Φ continuity means that an artifact remains linked to an opaque lineage without exposing the content of that lineage.

### 7.2 Sufficient Basis for Continuity Claims

A continuity claim SHOULD rely on at least one of:
- direct `ΦRef` continuity,
- `ΦHash` continuity plus provenance continuity,
- explicit continuity-proof artifacts,
- or an equivalent auditable lineage mechanism.

### 7.3 Hash Equality Is Not Sufficient by Itself

Matching `ΦHash` values alone MUST NOT be treated as universally sufficient proof of continuity.

Hash equality MAY contribute to a continuity claim, but SHOULD be evaluated together with:
- provenance context,
- issuance scope,
- and anti-rebinding guarantees where applicable.

### 7.4 Continuity Break

If an implementation cannot establish valid Φ continuity for an output artifact, then:
- the operation MUST fail,
- or the output MUST be explicitly marked non-conformant.

Silent loss of continuity is non-conformant.

## 8. Operator-Specific Φ Obligations

Every RFC-Λ operator MUST preserve Φ continuity.

### 8.1 Α (Alpha)

`Α` MUST:
- accept a `ΦToken` as part of `Vacuum`,
- produce a `ΦRef` or implementation-equivalent continuity artifact for the created `Node`,
- and preserve Φ without inspection.

Minimum required relation:

```text
Node.phi_ref = link(Vacuum.phi)
```

or an auditable equivalent.

---

### 8.2 Λ (Lambda)

`Λ` MUST:
- preserve the origin Φ lineage across all generated steps,
- ensure that no step breaks continuity,
- and avoid any policy or candidate-selection logic based on Φ internal content.

A `Trajectory` MUST be Φ-continuous with its origin `Node`.

---

### 8.3 Σ (Sigma)

`Σ` MUST:
- preserve all distinct source Φ lineages,
- aggregate them without collapsing them into one interpreted semantic object,
- and ensure that `Composite.phi_refs` or an equivalent structure retains plurality of source continuities.

Lossy union of multiple Φ lineages is non-conformant unless explicitly specified by a future profile RFC.

---

### 8.4 Ω (Omega)

`Ω` MUST:
- preserve the Φ continuity of its target artifact in the resulting `Closure`,
- through provenance, explicit references, or continuity proofs,
- without exposing Φ-content.

This RFC does not require a top-level `phi_ref` field in `Closure`, but continuity MUST remain auditable.

---

### 8.5 ∇ (Nabla)

`∇` MUST:
- preserve Φ continuity into persistent record lineage,
- ensure that persistence-facing material remains Φ-safe,
- and prohibit raw Φ exposure in:
  - `Record.index`
  - `Record.delta`
  - retrieval-visible material
  - and storage-visible query payloads

`∇` is the final L1 barrier before long-lived persistence. It is therefore a critical enforcement point.

---

## 9. Provenance Requirements

### 9.1 Provenance Binding

Every successful operator application involving Φ-bearing artifacts MUST extend provenance with sufficient material to support:
- lineage continuity,
- operator identity,
- actor identity,
- timestamp,
- and continuity-safe references or hashes where policy permits.

### 9.2 Provenance Content Rules

Provenance MAY contain:
- `ΦRef`
- `ΦHash`
- continuity proof references
- continuity validation results

Provenance MUST NOT contain:
- raw `ΦToken` content
- interpreted Φ-derived content

### 9.3 Provenance Monotonicity

Φ-related provenance MUST remain monotonic across the RFC-Λ cycle.  
A later artifact MUST NOT have less continuity evidence than is necessary to reconstruct its lineage under implementation rules.

## 10. Audit Requirements

### 10.1 Audit Minimum

A compliant implementation SHOULD emit Φ-related audit fields for each successful operator transition, including at minimum:

- `phi_preserved: Bool`
- `phi_continuity_claim: Bool`
- `phi_leak_detected: Bool`

### 10.2 Recommended Audit Fields

Where policy permits, implementations SHOULD also log:
- `phi_hash`
- continuity proof reference
- enforcement mechanism id
- leak detector status

### 10.3 Audit Safety

Audit logs MUST remain Φ-safe:
- raw Φ content MUST NOT appear,
- debug stringification of `ΦToken` MUST NOT be logged,
- and continuity metadata MUST remain non-semantic.

## 11. Leakage Model

### 11.1 Leakage Classes

This RFC defines the following leakage classes.

#### 11.1.1 Direct Content Leakage
Any direct exposure of raw `ΦToken` content.

#### 11.1.2 Derived Semantic Leakage
Any L1-visible field, decision, or artifact explicitly derived from Φ internal content.

#### 11.1.3 Control-Flow Leakage
Any branching of operator logic on Φ internal content.

#### 11.1.4 Metadata Leakage
Any unauthorized exposure through:
- logs,
- indexes,
- telemetry,
- analytics,
- profiling,
- or exported metadata.

#### 11.1.5 Persistence Leakage
Any storage-layer exposure that makes Φ-content or unauthorized semantic surrogates durable and retrievable.

#### 11.1.6 Side-Channel Leakage
Any implementation path that intentionally or negligently allows Φ-sensitive information to be inferred through:
- timing,
- size,
- memory layout,
- or access patterns,
where such leakage becomes part of L1-observable behavior.

### 11.2 Leakage Detection Expectations

A compliant implementation SHOULD implement at least one of:
- static checks,
- runtime taint tracking,
- audit anomaly detection,
- restricted serialization interfaces,
- wrapper-based opaque types.

### 11.3 Leakage Response

If a Φ leak is detected, the implementation MUST:

1. fail the current operation with an explicit Φ-related error;
2. emit a high-severity audit event if audit infrastructure is available;
3. avoid persisting any artifact contaminated by leaked Φ content;
4. avoid marking the operation as successful.

## 12. Failure Conditions and Error Semantics

A compliant implementation MUST provide explicit failure signaling.

### 12.1 Required Error Classes

At minimum, implementations MUST distinguish the following error categories:

- `PHI_PRECONDITION_FAILED`
- `PHI_CONTINUITY_BROKEN`
- `PHI_NON_OPAQUE_IMPLEMENTATION`
- `PHI_FORBIDDEN_OPERATION`
- `PHI_LEAKAGE_DETECTED`
- `PHI_AUDIT_WRITE_FAILED`
- `PHI_PROVENANCE_INVALID`

### 12.2 Failure Semantics

If Φ safety cannot be guaranteed for a transition:
- the transition MUST fail,
- or the produced artifact MUST be explicitly marked non-conformant and MUST NOT be treated as RFC-Λ compliant output.

Silent best-effort degradation is non-conformant.

### 12.3 Leakage Is Not Recoverable In-Place

If a raw Φ leak has occurred, the implementation MUST NOT simply continue as if the artifact were still clean.  
The contaminated execution path MUST be treated as compromised.

## 13. Conformance Requirements

### 13.1 Level Φ-A: Basic Opaque Preservation

A Level Φ-A implementation MUST:
- represent `ΦToken` as opaque,
- forbid all operations listed in Section 6.3,
- preserve Φ continuity across all RFC-Λ operators,
- extend provenance appropriately,
- and produce Φ-safe audit output.

### 13.2 Level Φ-B: Enhanced Continuity Assurance

A Level Φ-B implementation MUST additionally:
- support `ΦHash`,
- support explicit continuity validation,
- detect at least one leakage class automatically,
- and include continuity-safe metadata in provenance or audit.

### 13.3 Level Φ-C: High-Assurance Opaque Preservation

A Level Φ-C implementation MUST additionally:
- provide stronger evidence of non-inspection for critical paths,
- implement multiple leakage-detection controls,
- and support externally auditable continuity assurance mechanisms.

This RFC does not mandate zero-knowledge proofs or formal verification, but such mechanisms MAY be used to satisfy Level Φ-C requirements.

## 14. Security Considerations

### 14.1 Φ Is a Security Boundary

Even though Φ is not a conventional secret in all deployments, its non-inspectability is a security boundary for RFC-Λ conformance.

### 14.2 Side-Channel Risk

Implementations that treat Φ as opaque at the type level but leak it through timing, serialization, or logging remain non-conformant in practice.

### 14.3 Hash Security

If `ΦHash` is used, the hash function SHOULD be collision-resistant and appropriate to the threat model.

### 14.4 Provenance Forgery

If continuity claims depend on provenance, provenance integrity becomes security-critical.

### 14.5 Tooling Risk

Developer tooling, observability stacks, and debug middleware are common sources of accidental Φ leakage.

## 15. Ethics Considerations

This document standardizes only the L1 contract of opaque preservation.

However, Φ has direct ethical significance at the architectural level.

A system that violates Φ:
- collapses opacity into inspectability,
- reduces alterity to observable mechanics,
- and removes a minimal formal safeguard against total reduction.

Therefore:

1. **Opacity is not treated as an implementation accident.**  
   It is a protected architectural property.

2. **Non-inspection is not only technical hygiene.**  
   It is a precondition for systems that claim to preserve irreducibility.

3. **A fully transparent subject-model is outside the RFC-Λ design philosophy.**  
   This RFC formalizes that limit rather than denying it.

This document does not claim that Φ-preservation alone makes a system ethical.  
It does claim that violating Φ-preservation is incompatible with the minimal ethical architecture presupposed by the RFC-Λ family.

## 18. Appendix A: Correct and Incorrect Φ Handling

### A.1 Correct: Opaque Linking

```python
node.phi_ref = link(vacuum.phi)
```

### A.2 Correct: Integrity Check

```python
if phi_hash_a == phi_hash_b and provenance_chain_valid(a, b):
    continuity = True
```

### A.3 Incorrect: Raw Logging

```python
logger.debug("phi=%s", vacuum.phi)  # FORBIDDEN
```

### A.4 Incorrect: Branching on Internal Content

```python
if vacuum.phi.internal_flag == "special":   # FORBIDDEN
    policy = special_policy
```

### A.5 Incorrect: Search Index Exposure

```python
record.index["phi_payload"] = serialize(vacuum.phi)   # FORBIDDEN
```

### A.6 Incorrect: Semantic Rebinding via Hash

```python
if phi_hash in trusted_hashes:
    inferred_mood = "stable"   # FORBIDDEN
```

## 19. Appendix B: Informative Note on Formal Limits

RFC-Λ formalizes interfaces, transitions, invariants, and persistence behavior.  
It does not claim to exhaust what exists.

`Φ` is the explicit acknowledgement of that fact at the architectural level.

This is not an omission to be repaired later.  
It is a design limit intentionally preserved.

A compliant implementation therefore does not merely fail to interpret Φ.  
It is required not to pretend that such interpretation belongs to L1.
