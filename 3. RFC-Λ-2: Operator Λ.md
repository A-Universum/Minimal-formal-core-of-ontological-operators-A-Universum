## Abstract

This document specifies the formal L1 behavior of operator `Λ` (`Lambda`) within the RFC-Λ family.

`Λ` is the transition from `Node` to `Trajectory`. Formally, it constructs a temporally ordered sequence of boundary-compatible steps from a valid origin node under a declared policy. `Λ` is the primary operator of unfolding, traversal, and policy-governed continuation.

This document defines:
- the `Λ` operator signature,
- the policy contract,
- candidate generation requirements,
- trajectory construction semantics,
- status and halt semantics,
- provenance and audit obligations,
- and conformance requirements.

This document does **not** standardize:
- the full ontological meaning of becoming, dialogue, or path,
- the phenomenological quality of continuity or interruption,
- the final scoring semantics of all policy profiles,
- or higher-level modality systems beyond the minimal core contract.

Those concerns MAY be specified by profile-specific RFCs.

## 2. Conventions and Normative Language

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, **MAY**, and **OPTIONAL** are to be interpreted as described in RFC 2119 and RFC 8174.

This document inherits:
- the core type system from **RFC-Λ-0**,
- and the node semantics of **RFC-Λ-1**.

Unless otherwise specified:
- `Node`, `Step`, `Trajectory`, `Policy`, `Boundary`, `Context`, `ΦRef`, and `Provenance` have the meanings defined in RFC-Λ-0,
- and `Α` output semantics are those defined in RFC-Λ-1.

## 3. Operator Role

`Λ` is the primary traversal and unfolding operator of the RFC-Λ family.

At L1, `Λ` has exactly one role:

> Given a valid origin `Node`, a declared `Policy`, and a valid `Context`, construct a `Trajectory` as a temporally ordered sequence of boundary-compatible `Step` artifacts.

`Λ` does not:
- create a `Node` from `Vacuum`,
- compose multiple artifacts into a `Composite`,
- finalize an invariant into a `Closure`,
- or persist enriched records into storage.

Those functions belong to `Α`, `Σ`, `Ω`, and `∇`, respectively.

---

## 4. Motivation (Informative)

A marked node is not yet a trajectory. It is only the first admissible artifact in the operator chain.

`Λ` exists to make continuation formally tractable:
- traversal,
- branching,
- suspension,
- bounded continuation,
- dead-end detection,
- and policy-governed search or unfolding.

At the implementation level, `Λ` may represent:
- conversational continuation from an anchor node,
- semantic expansion in a graph,
- policy-governed exploration of memory space,
- iterative refinement of a state under admissibility constraints,
- or any other structured continuation process compatible with the core contract.

This document standardizes the formal contract of continuation. It remains agnostic regarding the full ontological meaning of path, resonance, tension, or pause.

---

## 5. Formal Signature

The formal signature of `Λ` is:

```text
Λ : Node × Policy × Context -> Trajectory × LambdaEffect
```

Where `LambdaEffect` is defined as:

```text
LambdaEffect = (
  status: TrajectoryStatus,
  steps_count: Integer,
  invariant_candidates: Set<ID> | Set<PatternRef> | InvariantSummary,
  omega_ready: Bool,
  termination_reason: TerminationReason
)
```

And:

```text
TerminationReason ::= completed | suspended | dead_end | limit | external_halt
```

### 5.1 Signature Interpretation

- `Node` is the origin artifact produced by `Α` or by an equivalent conformant constructor.
- `Policy` is the declared continuation profile.
- `Context` provides candidate generation, connector registry, actor identity, time, and storage.
- `Trajectory` is the ordered unfolding artifact.
- `LambdaEffect` is the observable formal effect.

## 6. Input and Output Types

### 6.1 Input: Node

The input `Node` MUST conform to RFC-Λ-0 and RFC-Λ-1.

For `Λ`, the following fields are REQUIRED:

```text
Node = (
  id: ID,
  fields: FieldMap,
  boundary: Boundary,
  phi_ref: ΦRef,
  provenance: Provenance,
  status: NodeStatus
)
```

A `Λ` invocation MUST treat the input node as the origin of the trajectory.

### 6.2 Input: Policy

The `Policy` type is defined in RFC-Λ-0:

```text
Policy = (
  name: Name,
  next: SelectionFunction,
  halt: HaltFunction
)
```

Where:

```text
SelectionFunction : Step × Set<FieldMap> × Context -> FieldMap | SUSPEND | HALT | NONE
HaltFunction      : Trajectory × Context -> Bool
```

For `Λ`, the selected policy MUST remain fixed for the duration of a single invocation unless a higher-level profile RFC explicitly defines dynamic policy-shift semantics.

### 6.3 Input: Context

For `Λ`, `Context` MUST provide at minimum:

1. actor identity,
2. time source,
3. audit-compatible store,
4. candidate generation support,
5. connector identification support,
6. optional invariant-candidate detection support.

### 6.4 Output: Trajectory

The output `Trajectory` MUST conform to RFC-Λ-0:

```text
Trajectory = (
  id: ID,
  origin: Node,
  steps: Sequence<Step>,
  policy: PolicyName,
  status: TrajectoryStatus,
  provenance: Provenance
)
```

### 6.5 Output: LambdaEffect

The operator MUST return a `LambdaEffect`.

If invariant candidates cannot be detected, the implementation MUST still return:
- a valid `status`,
- a valid `steps_count`,
- a valid `termination_reason`,
- and an empty or deferred invariant candidate representation.

## 7. Policy Contract

`Λ` requires a declared continuation policy.

### 7.1 Policy Identity

Each policy MUST have:
- a unique name within implementation scope,
- a stable version or profile identifier,
- auditable provenance.

### 7.2 Selection Function Requirements

A compliant `SelectionFunction` MUST:
1. accept the current `Step`, the candidate set, and `Context`,
2. return exactly one of:
   - a selected `FieldMap`,
   - `SUSPEND`,
   - `HALT`,
   - or `NONE`,
3. terminate or fail explicitly,
4. not inspect Φ-content,
5. not mutate historical steps.

### 7.3 Halt Function Requirements

A compliant `HaltFunction` MUST:
1. accept the current `Trajectory` and `Context`,
2. return a boolean,
3. terminate or fail explicitly,
4. not inspect Φ-content.

### 7.4 Heuristic Policies

Policies MAY be heuristic.

If heuristic, the implementation SHOULD:
- declare the policy as heuristic in metadata,
- version the heuristic logic,
- and preserve policy identity in audit records.

### 7.5 Standard Policy Names

This RFC does not standardize final semantics of named profiles such as:
- `rho`
- `sigma`
- `tau`

However, implementations MAY expose them as registered policies, provided they satisfy the core `Policy` contract.

Profile-specific RFCs MAY later define stronger semantics for those names.

---

## 8. Candidate Generation Contract

`Λ` requires a candidate generation mechanism.

### 8.1 Candidate Generator Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
generate_candidates : Step × Context -> Set<FieldMap>
```

### 8.2 Candidate Generator Requirements

The generator MUST:
1. terminate or fail explicitly,
2. return zero or more candidate `FieldMap` objects,
3. not mutate the current step,
4. not inspect Φ-content,
5. be auditable by generator identity or profile.

### 8.3 Boundary Filtering

Before policy selection, `Λ` MUST filter generated candidates through the origin boundary, unless a higher-level RFC explicitly defines envelope-based or transformed-boundary semantics.

Formally:

```text
ValidCandidates = { c | c ∈ generate_candidates(current, context) and origin.boundary(c) = true }
```

Only `ValidCandidates` MAY be passed to the policy selection function.

### 8.4 Empty Candidate Set

If `ValidCandidates` is empty:
- the selection function MAY still be invoked with an empty set,
- or the implementation MAY terminate immediately.

In either case, the resulting status and termination reason MUST be explicit and auditable.

## 9. Trajectory Construction Semantics

### 9.1 Informal Summary

`Λ`:
1. validates input admissibility,
2. creates an origin step from the origin node,
3. repeatedly generates valid candidate configurations,
4. applies the declared policy,
5. creates new steps while boundary validity is maintained,
6. terminates by halt, suspension, dead-end, or limit,
7. emits provenance and audit records.

### 9.2 Origin Step

A successful `Λ` invocation MUST create an origin step:

```text
Step_0 = (
  id = new ID,
  fields = origin.fields,
  boundary = origin.boundary,
  predecessor = null,
  connector = null,
  phi_ref = origin.phi_ref,
  timestamp = now
)
```

The origin step MUST be the first step in the trajectory.

### 9.3 Iterative Transition Rule

At iteration `i`, given current step `S_i`:

1. Generate candidates:
   ```text
   Candidates_i = generate_candidates(S_i, context)
   ```

2. Filter by origin boundary:
   ```text
   Valid_i = { c | c ∈ Candidates_i and origin.boundary(c) = true }
   ```

3. Apply policy selection:
   ```text
   Decision_i = policy.next(S_i, Valid_i, context)
   ```

4. Interpret decision:

- If `Decision_i` is a `FieldMap`, create `Step_{i+1}`.
- If `Decision_i = SUSPEND`, terminate with `status = suspended`.
- If `Decision_i = HALT`, terminate with `status = completed`.
- If `Decision_i = NONE`, terminate with `status = dead_end`, unless implementation defines an alternative valid dead-end mapping.

5. After each successful step append, evaluate:
   ```text
   policy.halt(trajectory, context)
   ```

If `true`, terminate with `status = completed`.

### 9.4 Step Construction

For each selected configuration `F_next`, `Λ` MUST create:

```text
Step_next = (
  id: new ID,
  fields: F_next,
  boundary: origin.boundary,
  predecessor: current.id,
  connector: identify_connector(current, F_next, context) | null,
  phi_ref: current.phi_ref,
  timestamp: now
)
```

### 9.5 Connector Identification

A compliant implementation SHOULD identify the connector class used for the transition.

If connector identification is unavailable, the implementation MAY store:
- `null`,
- or an implementation-defined unknown connector marker.

Such cases SHOULD be auditable.

### 9.6 Limit Semantics

An implementation MAY impose trajectory limits such as:
- max steps,
- time budget,
- memory budget,
- or platform-specific resource bounds.

If such a limit terminates the trajectory, the implementation MUST set:
- `status = limit`
- `termination_reason = limit`

## 10. Invariant Candidate Handling

`Λ` MAY detect invariant candidates during traversal.

### 10.1 Optionality

Invariant candidate detection is OPTIONAL at the core level.

A compliant implementation MAY return:
- exact candidate identifiers,
- pattern references,
- summaries,
- or an empty set.

### 10.2 Candidate Detector Interface

If implemented, invariant candidate detection SHOULD expose an auditable mechanism equivalent to:

```text
detect_invariant_candidates : Trajectory × Context -> Set<ID> | Set<PatternRef> | InvariantSummary
```

### 10.3 Omega Readiness

`LambdaEffect.omega_ready` MUST indicate whether the trajectory is formally eligible for `Ω` processing under implementation rules.

At minimum, a compliant implementation SHOULD set:

```text
omega_ready = true
```

only if:
- the trajectory has at least one step,
- the trajectory terminated with a non-running status,
- and the trajectory is durably auditable.

A profile-specific RFC MAY define stronger readiness conditions.

## 11. Status Semantics

The following `TrajectoryStatus` values are normative.

### 11.1 `running`

`running` MAY be used only during in-process execution. A returned trajectory MUST NOT have `running` status.

### 11.2 `suspended`

`suspended` means:
- continuation was intentionally paused,
- the trajectory is structurally valid,
- and may be resumed by profile-defined means outside this RFC.

### 11.3 `completed`

`completed` means:
- policy or halt logic declared termination,
- and the trajectory is structurally closed for this invocation.

`completed` does not imply ontological finality.

### 11.4 `dead_end`

`dead_end` means:
- no further admissible continuation was selected,
- and no successful continuation was constructed.

### 11.5 `limit`

`limit` means:
- the trajectory terminated due to implementation or policy-imposed bounds.

### 11.6 Termination Reason Mapping

A returned `LambdaEffect.termination_reason` MUST be consistent with `Trajectory.status`.

## 12. Preconditions

A compliant implementation MUST reject `Λ` application if any REQUIRED precondition fails.

### 12.1 Required Preconditions

#### Λ-P1: Origin node must be valid

The input artifact MUST be a valid `Node`.

#### Λ-P2: Origin node must carry a boundary

```text
origin.boundary
```

MUST exist and be callable or equivalent as a total predicate.

#### Λ-P3: Policy must be declared

A valid `Policy` MUST be supplied.

#### Λ-P4: Context must support candidate generation

The implementation MUST provide a candidate generation mechanism.

#### Λ-P5: Context must support audit write

A successful `Λ` invocation MUST be auditable.

#### Λ-P6: Φ continuity must be available

The origin node MUST contain a valid `phi_ref`.

### 12.2 Optional Profile Preconditions

Profiles MAY add further requirements, such as:
- policy review gates,
- actor authorization,
- domain-specific admissibility checks,
- bounded-depth requirements.

Such preconditions MUST be clearly marked as profile-specific.

---

## 13. Postconditions

A compliant implementation MUST satisfy the following postconditions after successful `Λ`.

### 13.1 Required Postconditions

#### Λ-Q1: Valid trajectory is returned

The returned artifact MUST be a valid `Trajectory`.

#### Λ-Q2: Origin step must exist

The trajectory MUST contain at least one step corresponding to the origin node.

#### Λ-Q3: Steps must be ordered

Steps MUST be temporally ordered and predecessor-consistent.

#### Λ-Q4: Boundary integrity must hold

Unless a higher-level RFC explicitly defines alternative semantics, every step in the trajectory MUST satisfy:

```text
origin.boundary(step.fields) = true
```

#### Λ-Q5: Φ continuity must hold

Every step MUST preserve valid Φ continuity through `phi_ref`.

#### Λ-Q6: Provenance must be extended

The returned trajectory MUST contain a new `Λ` provenance event.

#### Λ-Q7: Audit event must be emitted

A corresponding audit event MUST exist in the configured store.

#### Λ-Q8: LambdaEffect must be consistent with output

The returned `LambdaEffect` MUST accurately reflect:
- the trajectory status,
- step count,
- and termination reason.

### 13.2 Returned Status Constraint

A returned trajectory MUST have one of:

- `suspended`
- `completed`
- `dead_end`
- `limit`

A returned trajectory MUST NOT have `running` status.

## 14. Failure Conditions and Error Semantics

A compliant implementation MUST provide explicit failure signaling.

### 14.1 Required Error Classes

At minimum, implementations MUST distinguish the following error categories:

- `LAMBDA_PRECONDITION_FAILED`
- `LAMBDA_POLICY_INVALID`
- `LAMBDA_CANDIDATE_GENERATION_FAILED`
- `LAMBDA_BOUNDARY_VIOLATION`
- `LAMBDA_PHI_PRESERVATION_FAILED`
- `LAMBDA_AUDIT_WRITE_FAILED`
- `LAMBDA_LIMIT_EXCEEDED`

### 14.2 Failure Semantics

If `Λ` fails before trajectory creation:
- no successful `Trajectory` MUST be returned,
- no successful `LambdaEffect` MUST be returned.

If `Λ` fails after origin step creation but before durable completion:
- the implementation MUST either:
  - roll back entirely,
  - or persist an explicit failed-state artifact or failed audit event.

Silent partial success is non-conformant.

### 14.3 Boundary Violation Failure

If a step candidate selected by the policy does not satisfy the governing boundary and no higher-level RFC permits such transition, the implementation MUST:
- reject the step,
- and either continue selection or fail explicitly.

It MUST NOT silently append a boundary-violating step.

## 15. Φ-Preservation Requirements

`Λ` extends the Φ continuity established by `Α`.

### 15.1 Allowed Φ Operations in Λ

A compliant implementation MAY:
- carry forward Φ references,
- verify Φ continuity,
- log Φ continuity proofs,
- hash or compare Φ references where policy permits.

### 15.2 Forbidden Φ Operations in Λ

A compliant implementation MUST NOT:
- inspect Φ-content,
- branch policy logic on Φ internal content,
- serialize Φ-content into steps, provenance, or audit records.

### 15.3 Φ Continuity Requirement

For each appended step:

```text
step_i.phi_ref
```

MUST be continuity-preserving with respect to the origin node `phi_ref` and previous valid step, according to implementation Φ reference semantics.

## 16. Provenance and Audit Requirements

### 16.1 Provenance Events

A successful `Λ` invocation MUST append provenance information including at least:

- `operator = Λ`
- actor id
- timestamp of invocation
- origin node id
- policy name
- final trajectory status
- termination reason

An implementation MAY record per-step provenance events or a summarized event, provided auditability is preserved.

### 16.2 Audit Event

A successful `Λ` audit event MUST include at least:

- operator name
- trajectory id
- origin node id
- actor id
- timestamp
- policy name
- status
- termination reason
- step count

### 16.3 Per-Step Auditing

Implementations SHOULD support per-step audit visibility, especially where:
- policy heuristics are non-trivial,
- candidate generation is stochastic,
- or trajectory behavior affects durable memory or decision logic.

### 16.4 Append-Only Requirement

`Λ` audit records MUST be append-only or functionally equivalent under immutable versioning.

## 17. Conformance Requirements

### 17.1 Level A: Structural Conformance

A Level A `Λ` implementation MUST:
- accept the core signature,
- validate required preconditions,
- create a valid `Trajectory`,
- preserve Φ continuity,
- extend provenance,
- emit an audit event.

### 17.2 Level B: Semantic Conformance

A Level B `Λ` implementation MUST additionally:
- create an origin step from the origin node,
- enforce boundary integrity on appended steps,
- return a valid non-running terminal status,
- distinguish termination reasons,
- provide explicit error semantics.

### 17.3 Level C: Profile Conformance

A Level C `Λ` implementation MUST additionally conform to one or more registered policy profiles, such as:
- `rho`
- `sigma`
- `tau`

Profile conformance MUST NOT violate Level A or B requirements.

This RFC does not yet define canonical scoring semantics for those profiles.

## 18. Security Considerations

### 18.1 Policy Manipulation

A malicious or biased policy can distort traversal while remaining formally valid. Implementations SHOULD version, identify, and audit all policies.

### 18.2 Candidate Generator Abuse

A compromised candidate generator can constrain or poison the search space. Generator identity and version SHOULD be auditable.

### 18.3 Hidden Boundary Drift

If step construction silently changes boundary semantics, later operators become unreliable. Boundary identity SHOULD remain explicit in all steps.

### 18.4 Φ Leakage

Improper handling of Φ references during trajectory expansion can leak sensitive non-formalizable state. Debug output and serialization paths are common failure points.

### 18.5 Resource Exhaustion

Unbounded `Λ` execution can exhaust compute or memory resources. Implementations SHOULD provide resource bounds and explicit `limit` termination.

---

## 19. Ethics Considerations

This document standardizes only the L1 contract of `Λ`.

It does not imply that every formally valid continuation is ethically justified.

Implementers and deployers SHOULD consider:

1. A trajectory can encode asymmetry, exclusion, or escalation while remaining formally valid.
2. Policy identity matters ethically, not only technically.
3. Suspension (`suspended`) is not equivalent to resolution.
4. Dead-end detection does not imply impossibility outside the chosen policy or generator regime.
5. Long trajectories can accumulate institutional bias if persisted without review.

Profile-specific governance MAY be required for sensitive domains.

## 21. Appendix A: Canonical Minimal Example

### A.1 Input

```text
Origin Node:
  id = "node-001"
  fields = {topic: "session", stage: 1}
  boundary = B_origin
  phi_ref = "phi://ref-001"
  provenance = [...Α event...]
  status = open

Policy:
  name = "default-rho-like"
  next = policy.next(...)
  halt = policy.halt(...)

Context:
  actor = "operator-123"
  candidate generator = generate_candidates
  store = SemanticDB-compatible backend
  clock = ...
```

Assume:

```text
generate_candidates(step_0, context) returns:
  {
    {topic: "session", stage: 2},
    {topic: "other", stage: 99}
  }
```

and:

```text
B_origin({topic: "session", stage: 2}) = true
B_origin({topic: "other", stage: 99}) = false
```

### A.2 Output

```text
Trajectory:
  id = "traj-001"
  origin = node-001
  steps = [
    step-0: {topic: "session", stage: 1},
    step-1: {topic: "session", stage: 2}
  ]
  policy = "default-rho-like"
  status = completed
  provenance = [...Λ event...]

LambdaEffect:
  status = completed
  steps_count = 2
  invariant_candidates = {}
  omega_ready = true
  termination_reason = completed
```

### A.3 Required Truth Condition

For every step in the returned trajectory:

```text
B_origin(step.fields) = true
```

## 22. Appendix B: Informative Reference Pseudocode

The following pseudocode is informative only. It is not normative.

```python
def lambda_op(origin_node, policy, context):
    if origin_node is None:
        raise LambdaError("LAMBDA_PRECONDITION_FAILED")

    if origin_node.boundary is None:
        raise LambdaError("LAMBDA_PRECONDITION_FAILED")

    if policy is None or policy.next is None or policy.halt is None:
        raise LambdaError("LAMBDA_POLICY_INVALID")

    if origin_node.phi_ref is None:
        raise LambdaError("LAMBDA_PHI_PRESERVATION_FAILED")

    trajectory = Trajectory(
        id=context.new_id(),
        origin=origin_node,
        steps=[],
        policy=policy.name,
        status="running",
        provenance=context.extend_provenance(
            operator="Λ",
            actor=context.actor,
            origin=origin_node.id,
            policy=policy.name,
        ),
    )

    current = Step(
        id=context.new_id(),
        fields=origin_node.fields,
        boundary=origin_node.boundary,
        predecessor=None,
        connector=None,
        phi_ref=origin_node.phi_ref,
        timestamp=context.now(),
    )

    trajectory.steps.append(current)

    while True:
        candidates = context.generate_candidates(current, context)

        valid_candidates = {
            c for c in candidates
            if origin_node.boundary(c)
        }

        decision = policy.next(current, valid_candidates, context)

        if decision == "SUSPEND":
            trajectory.status = "suspended"
            termination_reason = "suspended"
            break

        if decision == "HALT":
            trajectory.status = "completed"
            termination_reason = "completed"
            break

        if decision in (None, "NONE"):
            trajectory.status = "dead_end"
            termination_reason = "dead_end"
            break

        if not origin_node.boundary(decision):
            raise LambdaError("LAMBDA_BOUNDARY_VIOLATION")

        next_step = Step(
            id=context.new_id(),
            fields=decision,
            boundary=origin_node.boundary,
            predecessor=current.id,
            connector=context.identify_connector(current, decision),
            phi_ref=current.phi_ref,
            timestamp=context.now(),
        )

        trajectory.steps.append(next_step)
        current = next_step

        if context.limit_reached(trajectory):
            trajectory.status = "limit"
            termination_reason = "limit"
            break

        if policy.halt(trajectory, context):
            trajectory.status = "completed"
            termination_reason = "completed"
            break

    if trajectory.status == "running":
        raise LambdaError("LAMBDA_INTERNAL_INVALID_RETURN_STATE")

    invariant_candidates = context.detect_invariant_candidates(trajectory) \
        if context.supports_invariant_detection() else set()

    omega_ready = (
        trajectory.status in {"completed", "suspended", "limit"}
        and len(trajectory.steps) >= 1
    )

    context.store.audit(
        operator="Λ",
        output=trajectory.id,
        actor=context.actor,
        timestamp=context.now(),
        origin=origin_node.id,
        policy=policy.name,
        status=trajectory.status,
        termination_reason=termination_reason,
        steps_count=len(trajectory.steps),
    )

    return trajectory, LambdaEffect(
        status=trajectory.status,
        steps_count=len(trajectory.steps),
        invariant_candidates=invariant_candidates,
```
