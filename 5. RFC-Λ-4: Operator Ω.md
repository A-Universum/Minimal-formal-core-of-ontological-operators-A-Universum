## Abstract

This document specifies the formal L1 behavior of operator `Ω` (`Omega`) within the RFC-Λ family.

`Ω` is the transition from a valid target artifact to a `Closure`. Formally, it:
- selects, rejects, or provisionally accepts an invariant,
- constructs an immutable closure artifact,
- computes a summary hash,
- preserves provenance and Φ continuity,
- and records audit information.

This document defines:
- the `Ω` operator signature,
- admissible target artifact classes,
- the `InvariantSelector` contract,
- closure status semantics,
- proof class handling,
- idempotence requirements,
- provenance and audit obligations,
- and conformance requirements.

This document does **not** standardize:
- the final ontological meaning of completion, return, or truth,
- the full semantics of all invariant scoring systems,
- or the philosophical adequacy of any extracted invariant.

Those concerns MAY be specified by profile-specific RFCs.

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, **MAY**, and **OPTIONAL** are to be interpreted as described in RFC 2119 and RFC 8174.

This document inherits:
- the core type system from **RFC-Λ-0**,

- the trajectory semantics of **RFC-Λ-2**,
- and the composite semantics of **RFC-Λ-3**.

Unless otherwise specified:
- `Trajectory`, `Composite`, `Invariant`, `Closure`, `Context`, `ΦRef`, `Hash`, and `Provenance` have the meanings defined in RFC-Λ-0,
- and `Α`, `Λ`, and `Σ` output semantics are those defined in RFC-Λ-1, RFC-Λ-2, and RFC-Λ-3.

## 3. Operator Role

`Ω` is the primary closure operator of the RFC-Λ family.

At L1, `Ω` has exactly one role:

> Given a valid target artifact and a declared `InvariantSelector`, construct an immutable `Closure` that records the invariant decision, summary hash, proof classification, provenance, and status.

`Ω` does not:
- create a `Node` from `Vacuum`,
- unfold a trajectory,
- compose multiple artifacts into a composite,
- or persist enriched records into storage.

Those functions belong to `Α`, `Λ`, `Σ`, and `∇`, respectively.

---

## 4. Motivation (Informative)

A trajectory may continue indefinitely. A composite may remain open and relation-bearing without yielding a stable conclusion. Systems require a formal mechanism for:
- termination,
- extraction,
- closure,
- explicit non-closure,
- and the recording of what has been selected as stable enough to carry forward.

`Ω` exists to formalize that step.

At the implementation level, `Ω` may represent:
- extraction of a stable pattern from a trajectory,
- closure of a composition around a selected invariant,
- production of a finalized summary for later persistence,
- or the formal declaration that no acceptable closure is currently available.

This document standardizes the formal contract of closure and invariant selection. It remains agnostic regarding whether any given closure is philosophically final, complete, or “true.”

---

## 5. Formal Signature

The formal signature of `Ω` is:

```text
Ω : TargetArtifact × InvariantSelector × Context -> Closure × OmegaEffect
```

Where:

```text
TargetArtifact ::= Trajectory | Composite
```

And `OmegaEffect` is defined as:

```text
OmegaEffect = (
  closed: Bool,
  invariant_id: ID,
  proof_class: ProofClass,
  summary_hash: Hash,
  closure_status: ClosureStatus,
  selector_decision: SelectorDecisionTag
)
```

Where:

```text
ClosureStatus ::= accepted | rejected | provisional
SelectorDecisionTag ::= accept | reject | provisional
```

### 5.1 Signature Interpretation

- `TargetArtifact` is the artifact being evaluated for closure.
- `InvariantSelector` is the decision-producing selector component.
- `Context` provides actor identity, time, store access, and summary-hash support.
- `Closure` is the resulting immutable closure artifact.
- `OmegaEffect` is the observable formal effect.

## 6. Admissible Target Artifact Classes

For the purposes of this RFC, the admissible input domain is:

```text
TargetArtifact ::= Trajectory | Composite
```

### 6.1 Base Exclusions

This RFC does **not** permit direct `Ω` closure over:
- `Node`
- `Closure`
- `Record`

A future profile-specific RFC MAY define:
- `Node`-level closure,
- re-closure over previous closures,
- or `Record`-derived closure semantics.

Such extensions MUST be explicit and versioned.

### 6.2 Advisory Nature of Prior Readiness Flags

Signals such as:
- `omega_ready` from RFC-Λ-2,
- `envelope_ready` from RFC-Λ-3,

MAY be used by implementations as advisory inputs, but MUST NOT be treated as universal preconditions in the base profile unless explicitly required by a profile-specific RFC.

## 7. Input and Output Types

### 7.1 Input: TargetArtifact

The target artifact MUST be valid in its declared class:
- `Trajectory`
- or `Composite`

It MUST provide:
- a stable identifier,
- provenance,
- and valid inherited Φ continuity.

### 7.2 Input: InvariantSelector

`InvariantSelector` is a required input to `Ω`.

A compliant selector MUST expose an L1-equivalent interface:

```text
InvariantSelector : TargetArtifact × Context -> SelectorDecision
```

`InvariantSelector` is specified in detail in [Section 8](#8-invariantselector-contract).

### 7.3 Input: Context

For `Ω`, `Context` MUST provide at minimum:

1. actor identity,
2. time source,
3. audit-compatible store,
4. summary-hash construction support,
5. selector identity support,
6. optional closure registry lookup support.

### 7.4 Output: Closure

The output `Closure` MUST conform to RFC-Λ-0:

```text
Closure = (
  id: ID,
  target_ref: ID,
  invariant: Invariant,
  summary_hash: Hash,
  provenance: Provenance,
  status: ClosureStatus
)
```

### 7.5 Output: OmegaEffect

The operator MUST return an `OmegaEffect`.

The effect MUST be consistent with:
- the closure status,
- the selector decision,
- the chosen invariant,
- and the generated summary hash.

## 8. InvariantSelector Contract

`Ω` requires an invariant decision mechanism.

### 8.1 Selector Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
InvariantSelector : TargetArtifact × Context -> SelectorDecision
```

Where:

```text
SelectorDecision ::= AcceptDecision | ProvisionalDecision | RejectDecision
```

### 8.2 AcceptDecision

```text
AcceptDecision = (
  tag: accept,
  invariant: Invariant
)
```

### 8.3 ProvisionalDecision

```text
ProvisionalDecision = (
  tag: provisional,
  invariant: Invariant
)
```

### 8.4 RejectDecision

```text
RejectDecision = (
  tag: reject,
  rejection_basis: RejectionBasis
)
```

Where `RejectionBasis` is implementation-defined but MUST be auditable.

### 8.5 Rejection Invariant Requirement

Because `Closure` in RFC-Λ-0 requires an `Invariant`, a rejected closure MUST still carry a valid invariant object in rejection form.

A rejection-form invariant MUST satisfy:

1. `proof_class = heuristic`
2. `support = ∅` or implementation-defined minimal rejection support
3. `schema` encodes rejection basis or rejection reference
4. it is explicitly marked as rejection-form in selector metadata or invariant metadata

This preserves totality of the `Closure` type.

### 8.6 Selector Requirements

A compliant `InvariantSelector` MUST:
1. terminate or fail explicitly,
2. return exactly one valid `SelectorDecision`,
3. not mutate the target artifact,
4. not inspect Φ-content,
5. be auditable by identity and version.

### 8.7 Determinism

An `InvariantSelector` SHOULD be deterministic for the same target artifact and profile version.

If nondeterministic, this MUST be declared in metadata and SHOULD be auditable.

## 9. Proof Class Semantics

This RFC uses the proof classes introduced in RFC-Λ-0:

- `heuristic`
- `reproducible`
- `strong`

### 9.1 `heuristic`

`heuristic` means:
- the invariant is accepted, provisional, or rejection-form under heuristic criteria,
- but no full reproducibility guarantee is claimed.

### 9.2 `reproducible`

`reproducible` means:
- the selector or profile claims repeatable extraction under declared conditions,
- and those conditions SHOULD be externally auditable.

### 9.3 `strong`

`strong` means:
- the selector or profile claims a stronger proof basis than mere reproducibility,
- and MUST provide explicit profile-level criteria for this claim.

### 9.4 Proof Class Integrity

A selector MUST NOT claim `reproducible` or `strong` without exposing auditable profile semantics sufficient to distinguish that claim from `heuristic`.

## 10. Closure Semantics

### 10.1 Informal Summary

`Ω`:
1. validates target admissibility,
2. invokes the selector,
3. normalizes selector output into a valid invariant-bearing closure,
4. computes a summary hash,
5. creates an immutable closure artifact,
6. extends provenance,
7. emits an audit event.

### 10.2 State Transition

Let:

```text
T = target artifact
D = selector(T, context)
```

Then `Ω(T, selector, context)` produces:

```text
C = (
  id,
  target_ref = id(T),
  invariant = I,
  summary_hash = H,
  provenance',
  status = S
)

E = (
  closed = true,
  invariant_id = id(I),
  proof_class = proof_class(I),
  summary_hash = H,
  closure_status = S,
  selector_decision = tag(D)
)
```

Where:

- if `D = AcceptDecision`, then `S = accepted`
- if `D = ProvisionalDecision`, then `S = provisional`
- if `D = RejectDecision`, then `S = rejected`

### 10.3 Closure Immutability

Once created, a `Closure` MUST be immutable.

Implementations MUST NOT modify:
- `target_ref`
- `invariant`
- `summary_hash`
- `status`

after successful durable creation, except by explicit supersession in a versioned higher-level mechanism outside this RFC.

### 10.4 Closure and Finality

`accepted` does not imply metaphysical finality.  
`provisional` does not imply invalidity.  
`rejected` does not imply target invalidity.

This RFC standardizes decision recording, not ultimate ontological truth.

## 11. Summary Hash Construction Contract

`Ω` requires a summary-hash mechanism.

### 11.1 Summary Hash Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
make_summary_hash : TargetArtifact × Invariant × InvariantSelector × Context -> Hash
```

### 11.2 Required Inputs to Summary Hash

A compliant summary hash MUST be derived from sufficient material to bind the closure to:

1. target identity,
2. invariant identity or invariant schema representation,
3. selector identity or selector profile identity,
4. closure status,
5. proof class.

### 11.3 Forbidden Inputs

A compliant summary hash MUST NOT include raw Φ-content.

Φ references MAY be included only as references or continuity proofs, not as inspected content.

### 11.4 Determinism

Given the same:
- target artifact,
- selector identity and version,
- invariant representation,
- closure status,
- proof class,

the summary hash SHOULD be deterministic.

If not deterministic, the implementation MUST declare that behavior explicitly.

## 12. Idempotence and Re-Invocation Semantics

### 12.1 Accepted Closure Idempotence

If a valid `accepted` closure already exists for:
- the same target artifact,
- the same selector identity and version,
- and the same summary-hash basis,

then re-invocation of `Ω` SHOULD return:
- the existing closure,
- or a reference-equivalent closure.

This is the base idempotence rule.

### 12.2 Provisional and Rejected Re-Invocation

For `provisional` and `rejected` closures, re-invocation semantics MAY vary by profile.

The base profile permits either:
- returning an existing equivalent closure,
- or generating a new closure if selector policy or target state has materially changed.

Such behavior SHOULD be auditable.

### 12.3 Idempotence Scope

Idempotence is scoped to:
- target identity,
- selector identity and version,
- and summary-hash basis.

Changes to any of these MAY legitimately produce a different closure.

## 13. Preconditions

A compliant implementation MUST reject `Ω` application if any REQUIRED precondition fails.

### 13.1 Required Preconditions

#### Ω-P1: Target artifact must be valid

The input artifact MUST be a valid `Trajectory` or `Composite`.

#### Ω-P2: Selector must be declared

A valid `InvariantSelector` MUST be supplied.

#### Ω-P3: Context must support summary hash construction

The implementation MUST provide summary-hash support.

#### Ω-P4: Context must support audit write

A successful `Ω` invocation MUST be auditable.

#### Ω-P5: Target provenance must be present

The target artifact MUST expose provenance or a valid provenance reference.

### 13.2 Optional Profile Preconditions

Profiles MAY define additional preconditions, such as:
- minimum step count,
- minimum relation count,
- minimum stability threshold,
- selector authorization,
- human review gate.

Such preconditions MUST be clearly marked as profile-specific.

---

## 14. Postconditions

A compliant implementation MUST satisfy the following postconditions after successful `Ω`.

### 14.1 Required Postconditions

#### Ω-Q1: Valid closure is returned

The returned artifact MUST be a valid `Closure`.

#### Ω-Q2: Closure must reference exactly one target

`closure.target_ref` MUST refer to the input target artifact.

#### Ω-Q3: Closure must contain exactly one invariant

The closure MUST contain one invariant, including rejection-form invariant where applicable.

#### Ω-Q4: Summary hash must exist

The closure MUST contain a valid `summary_hash`.

#### Ω-Q5: Closure status must be explicit

The closure MUST have one of:
- `accepted`
- `provisional`
- `rejected`

#### Ω-Q6: Provenance must be extended

The returned closure MUST contain a new `Ω` provenance event.

#### Ω-Q7: Audit event must be emitted

A corresponding audit event MUST exist in the configured store.

#### Ω-Q8: OmegaEffect must be consistent with output

The returned `OmegaEffect` MUST accurately reflect:
- closure status,
- selector decision,
- invariant identity,
- proof class,
- summary hash.

### 14.2 Immutability Constraint

A successfully created closure MUST be immutable as specified in [Section 10.3](#103-closure-immutability).

## 15. Failure Conditions and Error Semantics

A compliant implementation MUST provide explicit failure signaling.

### 15.1 Required Error Classes

At minimum, implementations MUST distinguish the following error categories:

- `OMEGA_PRECONDITION_FAILED`
- `OMEGA_SELECTOR_INVALID`
- `OMEGA_SELECTOR_EXECUTION_FAILED`
- `OMEGA_SUMMARY_HASH_FAILED`
- `OMEGA_PHI_PRESERVATION_FAILED`
- `OMEGA_AUDIT_WRITE_FAILED`
- `OMEGA_CLOSURE_REGISTRY_FAILED`

### 15.2 Failure Semantics

If `Ω` fails before closure creation:
- no successful `Closure` MUST be returned,
- no successful `OmegaEffect` MUST be returned.

If `Ω` fails after partial closure construction but before durable commit:
- the implementation MUST either:
  - roll back entirely,
  - or persist an explicit failed-state artifact or failed audit event.

Silent partial success is non-conformant.

### 15.3 Reject Is Not Failure

A selector decision of `reject` is **not** equivalent to operator failure.

A rejected closure is a valid successful output if:
- the target artifact was valid,
- the selector decision was valid,
- a rejection-form invariant was created,
- summary hash construction succeeded,
- provenance and audit obligations were met.

## 16. Φ-Preservation Requirements

`Ω` inherits Φ continuity from its target artifact.

### 16.1 Allowed Φ Operations in Ω

A compliant implementation MAY:
- carry forward target Φ references,
- verify Φ continuity,
- include Φ continuity proofs in audit material,
- use Φ references in provenance bindings.

### 16.2 Forbidden Φ Operations in Ω

A compliant implementation MUST NOT:
- inspect Φ-content,
- derive closure status from Φ internal content,
- serialize Φ-content into the closure, provenance, or audit records.

### 16.3 Φ Continuity Requirement

A closure MUST preserve the Φ continuity lineage of its target artifact via provenance and/or Φ reference continuity proofs.

This RFC does not require `Closure` to expose `phi_ref` as a top-level field; provenance-level continuity is sufficient.

## 17. Provenance and Audit Requirements

### 17.1 Provenance Events

A successful `Ω` invocation MUST append provenance information including at least:

- `operator = Ω`
- actor id
- timestamp of invocation
- target artifact id
- selector identity
- closure status
- proof class
- summary hash

### 17.2 Audit Event

A successful `Ω` audit event MUST include at least:

- operator name
- closure id
- actor id
- timestamp
- target id
- selector identity
- closure status
- proof class
- summary hash

### 17.3 Closure Registry Lookup

If an implementation supports closure registry lookup for idempotence, registry hits and misses SHOULD be auditable.

### 17.4 Append-Only Requirement

`Ω` audit records MUST be append-only or functionally equivalent under immutable versioning.

## 18. Conformance Requirements

### 18.1 Level A: Structural Conformance

A Level A `Ω` implementation MUST:
- accept the core signature,
- validate required preconditions,
- create a valid `Closure`,
- preserve target identity,
- preserve Φ continuity,
- extend provenance,
- emit an audit event.

### 18.2 Level B: Semantic Conformance

A Level B `Ω` implementation MUST additionally:
- implement explicit selector decision handling,
- support closure statuses `accepted`, `provisional`, and `rejected`,
- construct valid summary hashes,
- enforce closure immutability,
- provide explicit error semantics.

### 18.3 Level C: Profile Conformance

A Level C `Ω` implementation MUST additionally conform to one or more registered selector profiles, such as:
- heuristic selector profiles,
- reproducible selector profiles,
- strong-proof selector profiles,
- domain-specific selector profiles.

Profile conformance MUST NOT violate Level A or B requirements.

## 19. Security Considerations

### 19.1 Selector Manipulation

A malicious or biased selector can make unstable or politically convenient invariants appear closed. Selector identity and version SHOULD be auditable.

### 19.2 False Proof-Class Inflation

Incorrectly labeling a heuristic invariant as `reproducible` or `strong` is a major integrity risk. Implementations SHOULD log proof-class justification references.

### 19.3 Summary Hash Misbinding

If the summary hash does not bind target, invariant, selector, and status adequately, closure identity can be spoofed or confused.

### 19.4 Registry Poisoning

If closure lookup registries are compromised, false idempotence may result. Registry integrity SHOULD be protected.

### 19.5 Φ Leakage

Closure generation must not leak Φ-content through invariant extraction or summary-hash input paths.

---

## 20. Ethics Considerations

This document standardizes only the L1 contract of `Ω`.

It does not imply that every formally valid closure is ethically justified.

Implementers and deployers SHOULD consider:

1. Closure can turn provisional structure into durable authority.
2. A rejected closure may still have social consequences.
3. `accepted` can be mistaken for absolute truth if governance is weak.
4. `strong` proof-class claims can produce epistemic domination if used irresponsibly.
5. The ethics of ending a process are not reducible to hash stability.

Profiles MAY require human review, reversibility windows, or asymmetry audits for sensitive domains.

## 22. Appendix A: Canonical Minimal Example

### A.1 Input

```text
Target Artifact:
  Trajectory(id="traj-001", ...)

InvariantSelector:
  selector(target, context) returns:
    AcceptDecision(
      invariant = Invariant(
        id = "inv-001",
        schema = {topic: "session", stable_stage: 2},
        support = {"step-1", "step-2"},
        stability = 0.91,
        proof_class = reproducible
      )
    )
```

### A.2 Output

```text
Closure:
  id = "closure-001"
  target_ref = "traj-001"
  invariant = "inv-001"
  summary_hash = "hash://..."
  provenance = [...Ω event...]
  status = accepted

OmegaEffect:
  closed = true
  invariant_id = "inv-001"
  proof_class = reproducible
  summary_hash = "hash://..."
  closure_status = accepted
  selector_decision = accept
```

### A.3 Rejected Example (Minimal)

```text
Selector returns:
  RejectDecision(
    rejection_basis = "no invariant met stability threshold"
  )

Closure:
  status = rejected
  invariant = rejection-form invariant
```

This is a valid successful Ω output, not an operator failure.

## 23. Appendix B: Informative Reference Pseudocode

The following pseudocode is informative only. It is not normative.

```python
def omega(target, selector, context):
    if not context.is_valid_target_artifact(target):
        raise OmegaError("OMEGA_PRECONDITION_FAILED")

    if selector is None:
        raise OmegaError("OMEGA_SELECTOR_INVALID")

    # Optional idempotence lookup
    existing = context.lookup_closure(target, selector)
    if existing is not None and existing.status == "accepted":
        return existing, OmegaEffect(
            closed=True,
            invariant_id=existing.invariant.id,
            proof_class=existing.invariant.proof_class,
            summary_hash=existing.summary_hash,
            closure_status=existing.status,
            selector_decision="accept",
        )

    decision = selector(target, context)
    if decision is None:
        raise OmegaError("OMEGA_SELECTOR_EXECUTION_FAILED")

    if decision.tag == "accept":
        invariant = decision.invariant
        status = "accepted"

    elif decision.tag == "provisional":
        invariant = decision.invariant
        status = "provisional"

    elif decision.tag == "reject":
        invariant = context.make_rejection_invariant(decision.rejection_basis)
        status = "rejected"

    else:
        raise OmegaError("OMEGA_SELECTOR_INVALID")

    summary_hash = context.make_summary_hash(
        target=target,
        invariant=invariant,
        selector=selector,
        context=context,
        closure_status=status,
    )

    if summary_hash is None:
        raise OmegaError("OMEGA_SUMMARY_HASH_FAILED")

    closure = Closure(
        id=context.new_id(),
        target_ref=target.id,
        invariant=invariant,
        summary_hash=summary_hash,
        provenance=context.extend_provenance(
            operator="Ω",
            actor=context.actor,
            target=target.id,
            selector=context.selector_id(selector),
            closure_status=status,
            proof_class=invariant.proof_class,
            summary_hash=summary_hash,
        ),
        status=status,
    )

    context.store.audit(
        operator="Ω",
        output=closure.id,
        actor=context.actor,
        timestamp=context.now(),
        target=target.id,
        selector=context.selector_id(selector),
        closure_status=status,
        proof_class=invariant.proof_class,
        summary_hash=summary_hash,
    )

    return closure, OmegaEffect(
        closed=True,
        invariant_id=invariant.id,
        proof_class=invariant.proof_class,
        summary_hash=summary_hash,
        closure_status=status,
        selector_decision=decision.tag,
    )
```
