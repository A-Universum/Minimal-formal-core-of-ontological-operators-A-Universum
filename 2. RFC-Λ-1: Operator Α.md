## Abstract

This document specifies the formal L1 behavior of operator `Α` (`Alpha`) within the RFC-Λ family.

`Α` is the transition from `Vacuum` to `Node`. Formally, it introduces a marked artifact together with a boundary, provenance, and Φ-preserving reference. `Α` is the minimal constructive operator of the Λ system: it creates the first boundary-bearing artifact that can be processed by subsequent operators.

This document defines:
- the `Α` operator signature,
- input and output requirements,
- transition semantics,
- error conditions,
- provenance and audit obligations,
- and conformance requirements.

This document does **not** attempt to formalize the full ontological or phenomenological meaning of initiation, decision, or naming. Those remain outside L1 scope.

## 2. Conventions and Normative Language

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, **SHOULD NOT**, **MAY**, and **OPTIONAL** are to be interpreted as described in RFC 2119 and RFC 8174.

This document inherits all type definitions and global invariants from **RFC-Λ-0**.

Unless otherwise specified:
- `Vacuum`, `FieldMap`, `Boundary`, `Node`, `Context`, `ΦToken`, and `Provenance` have the meanings defined in RFC-Λ-0.
- `Α` denotes the operator specified in this document.
- `L0` refers to non-formalizable interpretation.
- `L1` refers to formal semantics.
- `L2` refers to implementation-specific code.

## 3. Operator Role

`Α` is the first constructive operator of the RFC-Λ family.

At L1, `Α` has exactly one role:

> Given an admissible `Vacuum`, a `FieldMap`, and a valid `Context`, construct a new `Node` with:
> - a unique identity,
> - a boundary,
> - a Φ-preserving reference,
> - a provenance entry,
> - and a corresponding audit record.

`Α` does not:
- create trajectories,
- compose multiple artifacts,
- finalize invariants,
- or persist enriched memory records.

Those functions belong to `Λ`, `Σ`, `Ω`, and `∇`, respectively.

---

## 4. Motivation (Informative)

In the RFC-Λ operator family, no higher-order operation is possible until a boundary-bearing artifact exists.

`Α` provides that artifact. It is therefore the minimal formal condition for:
- subsequent traversal (`Λ`),
- composition (`Σ`),
- closure (`Ω`),
- and enrichment (`∇`).

At the implementation level, `Α` corresponds to the creation of an initial marked object from an unmarked domain. This may represent, depending on system profile:
- the initialization of a semantic node,
- the formal marking of a conversational anchor,
- the creation of an entity in a memory graph,
- or the first stabilized distinction in a transformation pipeline.

This document remains agnostic regarding the ontological meaning of such acts. It standardizes only the executable contract.

---

## 5. Formal Signature

The formal signature of `Α` is:

```text
Α : Vacuum × FieldMap × Context -> Node × AlphaEffect
```

Where `AlphaEffect` is defined as:

```text
AlphaEffect = (
  created: Bool,
  collapsed_potential_ids: Set<ID> | PotentialSummary,
  phi_hash: Hash,
  collapse_mode: CollapseMode
)
```

And:

```text
CollapseMode ::= exact | partial | deferred | none
```

### 5.1 Signature Interpretation

- `Vacuum` is the unmarked input domain.
- `FieldMap` is the marking descriptor used to create the initial structured artifact.
- `Context` provides capabilities, time, actor identity, storage, and boundary construction support.
- `Node` is the newly created marked artifact.
- `AlphaEffect` is the observable formal effect.

---

## 6. Input and Output Types

### 6.1 Input: Vacuum

Input `Vacuum` MUST conform to RFC-Λ-0.

For `Α` application, the following fields are relevant:

```text
Vacuum = (
  potentials: Set<Potential> | Stream<Potential>,
  phi: ΦToken,
  unmarked: Bool
)
```

### 6.2 Input: FieldMap

`FieldMap` provides the initial marked content of the resulting node.

`FieldMap`:
- MAY be empty,
- MAY be sparse,
- MUST be syntactically valid for the implementation,
- MUST be acceptable to the boundary construction contract.

### 6.3 Input: Context

`Context` MUST provide:

1. actor identity,
2. capability inspection,
3. time source,
4. store access,
5. boundary construction support.

At minimum, the following capability is REQUIRED:

```text
node:create
```

### 6.4 Output: Node

The output `Node` MUST conform to RFC-Λ-0:

```text
Node = (
  id: ID,
  fields: FieldMap,
  boundary: Boundary,
  phi_ref: ΦRef,
  provenance: Provenance,
  status: NodeStatus
)
```

For `Α`, the created node status MUST be:

```text
open
```

unless a higher-level profile explicitly defines immediate closed-node semantics.

### 6.5 Output: AlphaEffect

The operator MUST return an `AlphaEffect`.

If exact collapse information cannot be computed, the implementation MUST still return:
- a valid `collapse_mode`,
- and a policy-compliant representation of collapse information.

---

## 7. Boundary Construction Contract

`Α` requires a boundary constructor.

### 7.1 Boundary Constructor Interface

A compliant implementation MUST provide an L1-equivalent mechanism:

```text
make_boundary : FieldMap × Context -> Boundary
```

### 7.2 Requirements on Boundary Constructor

The boundary constructor MUST:
1. terminate or fail explicitly,
2. return a total predicate over `FieldMap`,
3. be deterministic for the same `FieldMap` and boundary-construction profile, unless nondeterminism is explicitly declared,
4. be auditable by profile and version,
5. not inspect Φ-content.

### 7.3 Boundary Validity

A boundary is considered valid for `Α` if:
1. it is callable or equivalent as a predicate over `FieldMap`,
2. it can evaluate the created node’s own field map to `true`,
3. it can be serialized, hashed, or version-referenced for provenance purposes,
4. it does not violate implementation-defined safety constraints.

Formally:

```text
boundary(fields_created) = true
```

MUST hold.

### 7.4 Profile Sensitivity

Different implementations MAY use different boundary construction profiles.

Such profiles MUST be:
- versioned,
- identifiable,
- and auditable.

A profile-specific RFC MAY constrain boundary semantics further.

## 8. Operational Semantics

### 8.1 Informal Summary

`Α`:
1. validates input admissibility,
2. constructs a boundary,
3. creates a new node,
4. preserves Φ by reference only,
5. computes or summarizes collapsed potentials where possible,
6. records provenance and audit output.

### 8.2 State Transition

Let:

```text
V = (P, φ, true)
D = fields ∈ FieldMap
C = context
```

Then `Α(V, D, C)` produces:

```text
N = (id, D, B, phi_ref, provenance', open)
E = (true, collapsed, phi_hash, collapse_mode)
```

where:
- `B = make_boundary(D, C)`
- `phi_ref = link(φ)`
- `phi_hash = hash(φ)`

### 8.3 Collapsed Potentials

If `P` is finite and evaluable, then the exact collapsed set is:

```text
Collapsed(V, B) = { p.id | p ∈ P and B(p.delta) = false }
```

In this case:

```text
collapse_mode = exact
```

If `P` is not fully enumerable, the implementation MAY return:
- a partial set,
- a summary,
- or a deferred reference.

In those cases:
- `collapse_mode` MUST be `partial` or `deferred`,
- and the implementation MUST document the collapse policy.

If no potential information is available, then:
- `collapsed_potential_ids` MAY be empty,
- `collapse_mode` MUST be `none`.

### 8.4 Node Construction

The created node MUST satisfy:

1. `node.id` is unique in implementation scope,
2. `node.fields = D`,
3. `node.boundary = B`,
4. `node.phi_ref = link(φ)` or equivalent opaque reference,
5. `node.status = open`,
6. `node.provenance` contains at least one new `Α`-event.

### 8.5 Provenance Update

A successful `Α` application MUST append a provenance event with at least:

- operator identifier: `Α`
- actor identifier
- timestamp
- output node identifier
- boundary profile identifier or equivalent
- collapse mode

### 8.6 Audit Update

A successful `Α` application MUST append an audit event to the configured store.

The audit event MUST contain:
- operator name: `Α`
- output artifact id
- actor id
- timestamp
- success status
- collapse mode
- Φ continuity proof or hash reference

## 9. Preconditions

A compliant implementation MUST reject `Α` application if any REQUIRED precondition fails.

### 9.1 Required Preconditions

#### Α-P1: Vacuum must be unmarked

```text
vacuum.unmarked = true
```

MUST hold.

#### Α-P2: Context must permit node creation

`Context` MUST expose capability:

```text
node:create
```

#### Α-P3: ΦToken must be present

Input `Vacuum` MUST contain a valid `ΦToken`.

#### Α-P4: FieldMap must be syntactically valid

The supplied `FieldMap` MUST satisfy implementation syntax requirements.

#### Α-P5: Boundary constructor must succeed

`make_boundary(fields, context)` MUST either:
- return a valid boundary,
- or fail explicitly.

Silent fallback is non-conformant.

### 9.2 Optional Profile Preconditions

Profiles MAY define additional preconditions, such as:
- human review tokens,
- advocate approval,
- named policy gates,
- domain-specific capability checks.

Such profile preconditions MUST be clearly marked as profile-specific and MUST NOT be represented as universal `Α` core requirements unless adopted by a later revision of this RFC.

---

## 10. Postconditions

A compliant implementation MUST satisfy the following postconditions after successful `Α`.

### 10.1 Required Postconditions

#### Α-Q1: Valid node is returned

The returned artifact MUST be a valid `Node`.

#### Α-Q2: Node must carry a valid boundary

The node boundary MUST be callable or equivalent as a total admissibility predicate.

#### Α-Q3: Node must preserve Φ by reference

The output node MUST carry `phi_ref`, not Φ-content.

#### Α-Q4: Provenance must be extended

`node.provenance` MUST include a new event for this `Α` application.

#### Α-Q5: Audit event must be emitted

A corresponding audit event MUST exist in the configured store.

#### Α-Q6: AlphaEffect must be consistent with output

`AlphaEffect.created` MUST be `true` if and only if the node was created successfully.

### 10.2 Structural Consistency Condition

The created node MUST satisfy:

```text
node.boundary(node.fields) = true
```

This is REQUIRED for conformance.

## 11. Failure Conditions and Error Semantics

A compliant implementation MUST provide explicit failure signaling.

### 11.1 Required Error Classes

At minimum, implementations MUST distinguish the following error categories:

- `ALPHA_PRECONDITION_FAILED`
- `ALPHA_BOUNDARY_CONSTRUCTION_FAILED`
- `ALPHA_CONTEXT_CAPABILITY_DENIED`
- `ALPHA_PHI_PRESERVATION_FAILED`
- `ALPHA_AUDIT_WRITE_FAILED`

### 11.2 Failure Semantics

If `Α` fails before node creation:
- no `Node` MUST be returned,
- no successful `AlphaEffect` MUST be returned,
- a failure audit event SHOULD be emitted if infrastructure permits.

If `Α` fails after partial construction but before durable commit:
- the implementation MUST either roll back,
- or create an explicit failed-state record.

Silent partial success is non-conformant.

### 11.3 Recommended Error Structure

Implementations SHOULD expose machine-readable errors with:
- error code,
- error message,
- failed precondition or failed stage,
- timestamp,
- actor id where policy permits.

## 12. Φ-Preservation Requirements

`Α` is the first operator in the canonical sequence. It therefore establishes the initial Φ continuity contract.

### 12.1 Allowed Φ Operations in Α

A compliant implementation MAY:
- hash the input Φ token,
- create a link/reference,
- preserve the token opaquely,
- store a continuity proof.

### 12.2 Forbidden Φ Operations in Α

A compliant implementation MUST NOT:
- decode Φ-content into fields,
- derive boundary behavior from Φ internal content,
- log Φ-content,
- serialize Φ-content into provenance or audit structures.

### 12.3 Φ Continuity Requirement

The following relationship MUST hold:

```text
node.phi_ref = link(vacuum.phi)
```

or an implementation-equivalent continuity-preserving relation.

## 13. Provenance and Audit Requirements

### 13.1 Provenance Event

A successful `Α` provenance event MUST include at least:

- `operator = Α`
- `actor = context.actor`
- `timestamp = context.clock.now()`
- `output = node.id`
- `collapse_mode`
- `boundary_profile` or equivalent constructor identifier

### 13.2 Audit Event

A successful `Α` audit event MUST include at least:

- operator name
- output node id
- actor id
- timestamp
- success status
- Φ continuity proof reference
- collapse mode

### 13.3 Collapse Audit Semantics

If exact collapsed potentials are known, they SHOULD be auditable by identifier.

If exact collapsed potentials are unavailable, the audit event MUST declare:
- `collapse_mode = partial | deferred | none`

and SHOULD provide an implementation-defined explanation or reference.

### 13.4 Append-Only Requirement

`Α` audit records MUST be append-only or functionally equivalent to append-only under an immutable versioning discipline.

---

## 14. Conformance Requirements

### 14.1 Level A: Structural Conformance

A Level A `Α` implementation MUST:
- accept the core signature,
- validate required preconditions,
- create a valid `Node`,
- preserve Φ by reference,
- extend provenance,
- emit an audit event.

### 14.2 Level B: Semantic Conformance

A Level B `Α` implementation MUST additionally:
- provide a valid boundary constructor,
- guarantee `boundary(node.fields) = true`,
- distinguish collapse modes,
- provide explicit error semantics.

### 14.3 Level C: Profile Conformance

A Level C `Α` implementation MUST additionally conform to one or more registered `Α` profiles, such as:
- strict collapse profiles,
- advocate-gated profiles,
- bounded-potential profiles,
- streaming collapse profiles.

Profile conformance MUST NOT violate Level A or B requirements.

## 15. Security Considerations

### 15.1 Boundary Manipulation

A malicious or biased boundary constructor can distort all subsequent operations. Implementations SHOULD version and audit boundary construction profiles.

### 15.2 Φ Leakage

Because `Α` is the first preservation point, incorrect handling here compromises the entire operator chain. Implementations SHOULD isolate Φ processing and avoid debug exposure.

### 15.3 Collapse Misreporting

If collapsed potentials are reported inaccurately, provenance becomes misleading. Implementations SHOULD distinguish exact, partial, deferred, and absent collapse semantics explicitly.

### 15.4 ID Collision

Node identifiers MUST be unique in implementation scope. Collision resistance is REQUIRED for durable stores.

### 15.5 Incomplete Rollback

Partial construction without rollback or failed-state record creates integrity risks. Implementations MUST make failure states explicit.

---

## 16. Ethics Considerations

This document standardizes only the L1 contract of `Α`.

It does not imply that every formally valid act of marking is ethically justified.

Implementers and deployers SHOULD consider:

1. Boundary introduction is not ethically neutral.
2. A mark may exclude as well as define.
3. Reporting collapse incompletely may conceal asymmetry.
4. Automated marking of living, social, or sensitive systems may require external governance not expressible in this RFC.

Profiles MAY require human review or domain-specific authorization for those cases.

## 19. Appendix B: Informative Reference Pseudocode

The following pseudocode is informative only. It is not normative.

```python
def alpha(vacuum, fields, context):
    if vacuum.unmarked is not True:
        raise AlphaError("ALPHA_PRECONDITION_FAILED")

    if not context.has_capability("node:create"):
        raise AlphaError("ALPHA_CONTEXT_CAPABILITY_DENIED")

    if vacuum.phi is None:
        raise AlphaError("ALPHA_PHI_PRESERVATION_FAILED")

    boundary = context.make_boundary(fields, context)
    if boundary is None:
        raise AlphaError("ALPHA_BOUNDARY_CONSTRUCTION_FAILED")

    if not boundary(fields):
        raise AlphaError("ALPHA_BOUNDARY_CONSTRUCTION_FAILED")

    phi_ref = vacuum.phi.link()
    phi_hash = vacuum.phi.hash()

    collapsed = set()
    collapse_mode = "none"

    if is_finite(vacuum.potentials):
        collapsed = {
            p.id for p in vacuum.potentials
            if not boundary(p.delta)
        }
        collapse_mode = "exact"

    node = Node(
        id=context.new_id(),
        fields=fields,
        boundary=boundary,
        phi_ref=phi_ref,
        provenance=context.extend_provenance(
            operator="Α",
            actor=context.actor,
            boundary_profile=context.boundary_profile_id(),
            collapse_mode=collapse_mode
        ),
        status="open",
    )

    context.store.audit(
        operator="Α",
        output=node.id,
        actor=context.actor,
        timestamp=context.now(),
        success=True,
        phi_hash=phi_hash,
        collapse_mode=collapse_mode,
    )

    return node, AlphaEffect(
        created=True,
        collapsed_potential_ids=collapsed,
        phi_hash=phi_hash,
        collapse_mode=collapse_mode,
    )
```
